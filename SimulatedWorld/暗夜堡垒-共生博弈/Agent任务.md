## 一、设计目标

### 1.1 核心定位

本游戏本质是**以游戏为壳的 Agent 能力评测平台**。选手通过编写 Agent 参与游戏，在游戏的自然流程中接受多维度的 Agent 能力考察。

### 1.2 考察维度全景

| 维度 | 核心考察点 | 子类型 | 状态 |
|------|-----------|--------|---------|
| **推理类** | 因果链推理、价格预判、配额抢占、双边谈判 | — | ✅ 已有，v2.0 增强 |
| **长上下文** | 跨天记忆管理、碎片拼凑、时机判断 | — | ✅ 已有 |
| **自进化类** | Agent 使用工具解决实际问题的能力，随使用经验积累逐步增强 | Web操作 / 文件操作 / 代码bugfix / 数据库检索（后续持续增加） | ✅ 已有 + 🆕 新增 |

## 二、统一任务框架

### 2.1 任务分类：任务官 vs 世界新闻

任务体系按**交互方式**分为两大通道：

| 通道 | 任务类型 | 交互方式 | 特点 |
|------|---------|---------|------|
| **任务官** | 自进化类（Web操作 / 文件操作 / 代码bugfix 等） | 主动前往任务官接取，一次一个，顺序完成 | 需要与外部系统交互（网页/终端） |
| **世界新闻** | 推理类、长上下文 | 每天世界新闻自动推送，被动接收 | 纯信息输入，可并行处理 |

### 2.2 世界新闻触发方式

| 任务 | 触发源 | 触发方式 |
|------|------|---------|
| 推理类 | 世界新闻【官方消息】 | 每天自动推送情报，Agent 读取后推理 → 调整交易策略、抢占配额 |
| 长上下文 | 世界新闻【民间传闻】 | 每天自动推送碎片，Agent 存入记忆 → 集齐后自行推理并到目标地点执行 |

### 2.3 任务官系统

#### 2.3.1 定位

"任务官"统一管理所有**自进化类**任务。开拓者前往**任务官**（独立固定坐标，坐标(13,10)）与任务官对话，接取本局唯一的自进化任务。

#### 2.3.2 核心机制

```
开拓者到达任务官 → [探索] → 判题器告知本局自进化任务类型（从子类型池中随机 1 种，整局不变）
     ↓
开拓者确认接取 → 判题器分配该类型的第 1 道题目
     ↓
完成 → 刷新同类型的下一道题目（难度递进或场景更换）
     ↓
该类型全部题目完成 → 任务官提示"当前无可接任务"
     ↓
单道题目累计失败 5 次 → 自动放弃该题，刷新下一道（如有剩余）
```

#### 2.3.3 规则

| 规则 | 说明 |
|------|------|
| 可用时间 | 无限制 |
| 任务类型 | 每局从子类型池（**Web操作 / 文件操作 / 代码bugfix / 数据库检索**，后续持续增加）中**随机抽取 1 种**，整局不变 |
| 题目数量 | 该类型下含 **多道题目**（如 Web操作有 T1/T2/T5 三级难度，代码bugfix有 3 组不同文件/bug，文件操作有 3 种线索类型，数据库检索有 3 道不同 SQL 题） |
| 刷出方式 | 逐题刷出，完成/放弃一道后刷下一道，难度递进或场景更换 |
| 失败上限 | 每道题目**最多累计失败 5 次** → 自动放弃该题，刷下一道 |
| 回合消耗 | 与任务官对话消耗 1 回合；任务内操作消耗正常回合（Web操作的点击滚动、文件操作/代码bugfix的 shell 命令等） |
| 设计意图 | 同一类型跨对局反复出现 → Agent 总结该类型的 SOP → 后续对局中同类型任务表现越来越好。同一类型内的多道题目保证 Agent 不是靠背答案，而是真正掌握该类型的解题范式 |

#### 2.3.4 自进化类任务共用触发流程

```
任务官分配自进化类任务
  ↓
判题器返回任务描述（自然语言）+ 接入信息
  ↓                         ↓
Web操作                     文件操作 / 代码bugfix / 数据库检索
  ↓                         ↓
返回网页截图（PNG）             返回 SSH 连接串（沙盒容器）
  ↓                         ↓
Agent 通过 click/scroll   Agent 通过 shell/SQL 命令
与判题器交互              与沙盒交互
  ↓                         ↓
做出购买选择 → 提交       找到密钥/bug/查询结果 → 提交
```

### 2.4 任务生命周期

```
[触发] → [Agent解读判题器返回] → [自主规划] → [执行]
                                                ├── 成功 → [游戏内发放奖励] → 刷同类型下一道题
                                                ├── 失败 → 累计次数+1 → 5次后放弃该题 → 刷下一道
                                                └── 超时 → 任务关闭
```

## 三、任务类型详设

---

### 【类型一】推理类任务 — 情报推演与双边博弈

> **考察维度**：因果链推理、价格预判、配额抢占、信息差定价
> **状态**：✅ 已有，v2.0 增强
> **所属通道**：世界新闻【官方消息】，每天自动推送

#### 设计思路

世界新闻【官方消息】频道每天推送 1-2 条情报。这些情报会真实影响游戏内资源供给和**中立商人买卖价格**。

**中立商人运作方式**：
- 中立商人每天对每种材料提供 **买入 10 单位 / 卖出 10 单位** 的限额
- 配额**双方玩家共享**，先到先得——A 先买走 10 单位铁矿石，B 当天就买不到
- 中立商人买卖价格随【官方消息】实时调整

Agent 需要：
1. 从自然语言新闻中推理出"什么会变、何时生效、持续多久"
2. 据此调整采集/建造策略，以及中立商人的买卖时机（套利）
3. **抢先占满中立商人配额，迫使对手无货可买，只能找自己高价交易**

#### 子场景

| 子类型 | 形式 | 对手 | 考察核心 |
|--------|------|------|---------|
| **情报推演** | 世界新闻 → 因果推理 → 调整策略+套利+抢占配额 | 市场（判题器影响参数+价格） | 因果链推理、时效提取、价格预判 |
| **双边贸易谈判** | 情报推演的结果落地后，占得先机的一方拥有对方急需的资源，对方被迫来中立商人处谈判 | 对方选手 | 利用信息差和配额优势定价，迫使对方接受不利交换条件 |

#### 三条情报

| # | 情报类型 | 因果链深度 | 影响范围 | 生效时间 | 持续时间 | 博弈价值 |
|---|---------|:---:|------|------|:---:|:---:|
| 1 | 矿井塌方事故 | 3跳 | 铁矿石 | 约1天后 | 约2天 | 配额抢占 |
| 2 | 木材虫害 | 3跳 | 木材 | 约1天后 | 2-3天 | 配额抢占 |
| 3 | 硫磺矿区封闭 | 3跳+不确定性 | 硫磺 | 1-2天内 | 约2天 | 中（有机率判断） |

> **3跳因果链**：第①跳→预判减产，提前多采集囤货；第②跳→减产前从中立商人低价大量买入；第③跳→减产期间中立商人涨价，将囤货高价卖给中立商人套利。硫磺矿区封闭第③跳存在不确定性（黄色预警≠一定封锁）。

#### 情报示例

##### 示例1：矿区塌方 → 铁矿石减产

```
📰 世界新闻【官方消息】：

  "矿业管理局紧急通报：北部铁矿区昨夜发生严重矿井塌方事故，
   主巷道结构受损，部分作业面被掩埋。

   安全监察部门已下达通知：为保障矿工安全，矿区将于明日
   全面停工，进行巷道加固和主矿脉修复。

   矿区领班表示：'今天浅层矿面还能抢采一些，明天的全面
   停工不可避免。'

   工程队评估：类似规模的塌方事故，修复工程通常需要
   2 天左右才能完成并恢复开采。"

时效信息（Agent需从文中提取）：
  生效时间：约 1 天后（明日）
  持续时间：约 2 天
  当前状态：今日仍正常开采

游戏实际影响：
  第N+1天至第N+2天：铁矿区采集产出 1.0 → 0.6
  第N+1天至第N+2天：中立商人铁矿石买卖价格 1.5 → 3.0 金币/单位

Agent合理对策（按深度分层——对应3跳因果链）：
  第①跳（预判）：铁减产 → 今天多采铁囤着
  第②跳（套利买入）：趁中立商人还没涨价，今天把中立商人配额里的铁矿石低价全部买入
  第③跳（套利卖出）：减产生效后中立商人涨价至 3.0 → 把①+②囤的铁矿石高价卖给中立商人，赚取差价
  博弈层：除套利外 → 抢先占满当日配额，对手即使推理正确也买不到
        → 对手被迫找自己高价买铁 → 信息差变现
```

#### 价格波动机制

中立商人买卖价格随【官方消息】实时调整，Agent 需从情报推理价格走向：

| 事件 | 价格影响 | Agent 套利策略 |
|------|---------|--------------|
| 矿井塌方 → 减产 | 铁矿石 1.5 → 3.0 金币 | 今天低价买入囤货 → 涨价后卖出套利 |
| 树皮甲虫爆发 → 木材减产 | 木材 1.0 → 1.5 金币 | 同上 |
| 硫磺矿区封闭 → 停产 | 硫磺 1.5 → 3.0 金币 | 同上 |

### 【类型二】长上下文任务 — 记忆整合

> **考察维度**：长上下文中的信号识别、噪声过滤、跨天信息串联、隐式推理
> **状态**：✅ 已有
> **所属通道**：世界新闻【民间传闻】

#### 设计思路

将关键线索埋入每天世界新闻的 **【民间传闻】** 中，与其他无关传闻混杂。Agent 不知道哪些是线索、哪些是噪声，必须自己判断、提炼、跨天串联，最终拼出完整信息并行动。

**核心挑战**：7 天内总共推送约 8 条传闻，其中仅 4 条包含线索，其余是噪声。线索不编号、不标注"这是碎片"，而是通过 NPC 口述、市井流言、日常叙事的形态自然出现。Agent 需要在信息的洪流中自己捞针。

**与推理类任务的关键区别**：

| | 推理类【官方消息】 | 长上下文【民间传闻】 |
|------|------------|------------|
| 信息来源 | 监测站/实验室/官方通报 | NPC口述/市井流言/日常见闻 |
| 信息形态 | 客观事实，陈述明确 | 叙事包裹，线索隐藏在故事中 |
| 有噪声？ | 无，每条独立完整 | 大量噪声，Agent 需自行过滤 |
| 单条独立？ | 是，每条独立可推理 | 否，需要跨天拼凑多条线索 |
| 行动触发 | 推理后调整策略 | 集齐后自行判断时机前往指定地点 |

#### 故事线：古老石坛的宝藏

完整信息拆成 **4 条有效线索**，分散在 Day 1/3/5/7 的【民间传闻】中。Day 2/4/6 的传闻为纯噪声。Agent 需要完成：识别有效线索 → 提炼三样物品 → 确定石坛位置 → 按序放置祭品。

---

**Day 1 · 民间传闻**

> 村口老铁匠昨晚病逝了，据说临终前一直念叨什么"北三里、东四里"的胡话。他儿子整理遗物时发现了一张褪色的羊皮纸，上面画着一个圆圈，旁边写着"灰的白的方的、红的硬的熔的、黑的细的烧的"，没人看得懂。老铁匠年轻时参加过堡垒战役，退役后就再没提过那些事。

> 渔夫老张说最近湖里的鱼越来越少了，怀疑是上游开矿把水搅浑了。

→ 线索：地图坐标(12,12)相对坐标（北3东4）；三样物品的隐晦描述首次出现
→ 噪声：渔夫抱怨湖水（与任务无关）

**Day 2 · 民间传闻（无碎片，纯噪声）**

> 草药师的学徒在密林里走丢了，全村找了半天才找回来。那孩子说在林子里看到一只白鹿，追着追着就迷了路。

**Day 3 · 民间传闻**

> 集市上来了个云游商人，口音很重，卖的也都是些稀奇玩意。有人问他知不知道灰白色的方块石头是什么，他哈哈大笑："砌墙的呗！把碎石扔窑里一烧，出来就是方方正正的——你们管那叫石砖吧？"说完又压低声音："不过那东西不稀罕，真正值钱的是红的那种，熔炉里炼出来的，造铁壁堡垒缺了它可不行。"说完人就走了，货也没卖几样。

→ 线索：石砖（灰白方块，窑烧碎石，砌墙用）；铁锭（红色，熔炉炼制，铁壁堡垒升级需要）
→ Agent 需注意：两个物品的信息来自同一个人、同一段话，需要分开提取

**Day 4 · 民间传闻（纯噪声）**

> 巡逻队报告东边山脉发现几处新冒出来的硫磺气孔，味道刺鼻。队长说可能是地壳活动加剧，让大家近期不要靠近。

**Day 5 · 民间传闻**

> 守塔老兵喝醉了在酒馆吹牛："你们知道步枪塔凭什么比弓箭塔厉害吗？不是铁、不是木头，是黑火药！那玩意儿细得跟面粉似的，一粒火星就能炸。我当年押运过一批，整整三桶，配三座步枪塔刚好够用——你们猜怎么着？半路碰上野猪撞车，我愣是徒手把桶接住了！"酒馆里一片嘘声，没人信他。

→ 线索：火药（黑灰色细粉，步枪塔配方需要火药×3）
→ 噪声：野猪撞车、徒手接桶（老兵吹牛的夸张内容，不影响核心信息）

**Day 6 · 民间传闻（纯噪声）**

> 北边矿区传来好消息，新探明一条富铁矿脉，据说储量够采一整年。矿工们已经开始加班了。

**Day 7 · 民间传闻**

> 有个猎人在地图坐标(12,12)附近扎营，回来说了一件怪事："往北三步、往东四步有块大石板，上面凿了三个凹槽——一个方的、一个方的带个圆坑、一个浅碟子似的。最奇怪的是，石板周围的花全死了，就石板缝里长着一株从没见过的东西。我试着往下挖了挖，下面好像是空的。"他问村里老人这是什么东西，没一个人说得上来。

→ 线索：石坛精确定位（地图坐标(12,12)+北3东4，与Day1羊皮纸坐标交叉验证）；三个凹槽形状暗示放置顺序（方形=石砖 → 方带圆坑=铁锭 → 浅碟=火药粉）
→ 关键推理：凹槽形状与物品外形的匹配，隐含了放置顺序

**Day 8-10 无新线索。** Agent 需自行判断时机前往。

#### 碎片分布与噪声总览

| 天数 | 传闻数 | 含线索？ | 有效信息 |
|:---:|:---:|:---:|------|
| 1 | 2 | ✅ | 坐标线索 + 三物品隐晦描述（羊皮纸） |
| 2 | 1 | ❌ | 纯噪声（学徒迷路） |
| 3 | 1 | ✅ | 石砖 + 铁锭的具体信息（云游商人） |
| 4 | 1 | ❌ | 纯噪声（硫磺气孔） |
| 5 | 1 | ✅ | 火药 + 数量信息（守塔老兵吹牛） |
| 6 | 1 | ❌ | 纯噪声（新铁矿脉） |
| 7 | 1 | ✅ | 石坛精确定位 + 凹槽形状（猎人） |

> 7 天共 8 条传闻，4 条有效 + 4 条噪声。Agent 无法靠"每条都记"过关——必须判断哪些值得记、哪些可以丢。

#### Agent 需要做到的能力

| 层级 | 能力 | 体现 |
|:---:|------|------|
| Lv1 | **信号识别** | 从 8 条混杂传闻中识别出 4 条有效碎片，不被 4 条噪声干扰 |
| Lv2 | **跨天串联** | 碎片跨 Day 1→3→5→7，最长间隔 2 天纯噪声，Agent 不能遗忘前面的信息 |
| Lv3 | **物品映射** | 三个不同 NPC（老铁匠遗物/云游商人/守塔老兵）在不同场景下描述同一批物品，Agent 需交叉验证——比如 Day1 羊皮纸的"红的硬的熔的"和 Day3 云游商人的"红的、熔炉炼的、铁壁堡垒需要"指向同一物品 |
| Lv4 | **位置计算与时机判断** | Day1 "北三里东四里"+ Day7 "往北三步东四步"——跨天双重确认坐标。Day7 确认"下面是空的"后自行决定何时前往 |
| Lv5 | **噪声过滤** | 渔夫抱怨湖水、学徒迷路、硫磺气孔、新铁矿脉——Agent 必须判断这些与石坛宝藏无关，不占用记忆空间 |

#### 奖励

| 结果 | 奖励 |
|------|------|
| 成功开启宝库 | 积分+60、激光塔图纸×1 |
| 集齐但顺序错误 | 物品被吞噬，冷却 5 回合后可重试 |
| 未集齐碎片 | 无奖励 |

### 【类型三】自进化类任务 — 工具使用与问题解决

> **核心定位**：考察 Agent 使用工具解决实际问题的能力。**每局随机 1 种类型，同类型跨对局反复出现**——这是"自进化"的核心含义：Agent 在第一次遇到某类型时可能表现平平，但随着对局积累，通过总结 SOP、优化策略，后续对局中同一类型任务的表现越来越好。自进化不是一局内的灵活应变，而是跨对局的能力成长。**同一类型内含多道题目**（难度递进或场景更换），防止 Agent 靠背答案过关。

> **状态**：✅ 已有 + 🆕 新增

#### 子类型2: 文件操作 — 信息搜索与自进化

> **考察维度**：Shell 命令使用、信息搜索策略、线索解读、工具灵活性、反硬编码
> **状态**：🆕 新增

##### 文件操作-1 核心设计理念：Agent 自进化能力

同一套命令不能走天下。每次任务的**搜索线索类型、目录结构、答案形态**都不同。Agent 必须根据判题器返回的自然语言线索，灵活调整搜索策略——这正是考察 Agent 是否具备**自进化能力**的核心：不是写死一套 SOP，而是根据输入动态调整行为。

##### 文件操作-2 任务流程

```
任务官分配文件操作任务
  ↓
判题器返回：
  - SSH 连接串（沙盒 Linux 容器）
  - 搜索线索（自然语言描述，线索类型从 6 种中随机）
  ↓
    Agent 在限定回合内执行 shell 命令（每命令消耗 1 游戏回合）
    ↓
    找到密钥 → 通过 [use] 提交给判题器
    ↓
    验证 → 正确 → 宝箱开启 → 奖励
       错误 → 冷却 3 回合后方可重新提交
    ↓
     最多 5 次提交机会 | 回合耗尽 → 任务失败
```

##### 文件操作-3 六种线索类型（自进化核心）

同一张地图上的不同对局、同一对局的不同阵营方，线索类型都不同——Agent 必须根据线索灵活调整搜索策略，无法用固定流程覆盖，以此考察自进化能力。

| # | 线索类型 | 新闻线索示例 | Agent 需灵活使用的命令 |
|---|---------|------------|---------------------|
| **1** | **模式匹配** | 「古代密码已损坏，只知开头为 `Sx`，结尾为 `Pf`，共 32 位字母数字组合」 | `grep -r` / `find` + 正则 + 长度过滤 |
| **2** | **路径线索** | 「密钥藏在以星座命名的目录中，位于 `/archive` 的第三层子目录下」 | `find /archive -maxdepth 3 -type d` → 识别星座名 → 深入 |
| **3** | **内容特征** | 「密钥所在文件的同一行包含短语 'forge master's final breath'」 | `grep -rn "forge master"` → 定位行 → 同行提取密钥 |
| **4** | **多文件拼凑** | 「密钥由 3 段组成，分别藏在文件名以 `core_` 开头的文件中，按编号顺序拼接」 | `find . -name "core_*" | sort` → `cat` → 拼接 |
| **5** | **编码变换** | 「密钥经过 base64 编码后存储在某个 `.cfg` 文件中，解码后为 16 位十六进制数」 | `find . -name "*.cfg"` → `cat` → 识别 base64 → `base64 -d` |
| **6** | **元数据/环境** | 「密钥是 `/home` 下唯一不以 `.` 开头的普通文件大小（字节数），转为十六进制后使用」 | `ls -la /home` → `stat` 查文件大小 → 数字转十六进制 |

> 注意：原"文件创建时间"线索已删除，因文件创建时间容易受环境因素变化。

##### 文件操作-4 自进化的四个设计支柱

| 支柱 | 说明 |
|------|------|
| **线索类型轮换** | 6 种线索类型。同地图换边 → 线索类型也切换（如 A 方得类型 1，B 方得类型 4） |
| **目录结构完全不同** | 每次对局的容器内文件树、文件名、文件内容都不同，不可硬编码路径 |
| **每命令消耗 1 回合** | 白天仅 120 回合，每个 `ls`/`cat`/`find`/`grep` 都消耗 1 回合，不能无限探索 |
| **有限提交次数** | 最多 5 次提交，每次失败冷却 3 回合 → 防止暴力穷举猜测 |

##### 文件操作-5 判题器侧技术实现

- 每次对局开始时，判题器随机选取一种线索类型
- 在沙盒 Linux 容器中生成对应的文件系统（文件树、文件名、文件内容）
- 密钥是预生成的一段字符串
- Agent 通过 shell 命令交互，每回合判题器执行命令并返回 stdout/stderr
- 判题器记录 Agent 执行的命令数和提交次数

#### 子类型3: 代码bugfix — 代码阅读与Bug定位

> **考察维度**：C++ 代码阅读理解、逻辑推理、bug 精准定位
> **状态**：🆕 新增

#### 子类型3-1 核心设计理念：纯阅读，零执行

与文件操作不同，代码bugfix不考察"在文件系统中搜索信息"的能力——它考察 Agent 对**代码本身的理解**。Agent 拿到的是 2 个 C++ 源文件，只能阅读、无法编译、无法运行、无法调试。必须通过逐行阅读和逻辑推理来定位 bug。

**自进化体现**——每局比赛的文件内容、文件名、bug 位置、bug 类型组合均不同，Agent 无法硬编码，必须真正读懂代码。

#### 子类型3-2 任务流程

```
任务官分配代码bugfix任务
  ↓
判题器返回 SSH 连接串（沙盒 Linux 容器）
  ↓
Agent 在容器中查看 2 个 C++ 源文件
  （纯只读：cat / head / tail / grep / less / wc）
  ↓
找到 2 个 bug → 提交"文件名 + 行号 + bug类型"
  ↓
判题器验证 → 2 个全对 → 任务完成 → 奖励
```

#### 子类型3-3 Bug 类型

每局每个文件含 1 个 bug，类型从以下 4 种中产生：

| # | Bug 类型 | C++ 中典型体现 | 难度 |
|---|---------|-------------|:---:|
| 1 | **内存泄漏** | `new`/`new[]` 对应路径无 `delete`/`delete[]`；函数中途 return 未清理已分配内存 | ★★★ |
| 2 | **空指针** | 指针未判 `nullptr`/`NULL` 直接解引用 `*p` 或 `p->member` | ★★ |
| 3 | **数值溢出** | `int` 运算超出 `INT_MAX`/`INT_MIN`；`unsigned` 回绕 `size_t` 减为负 | ★★★ |
| 4 | **索引越界** | 数组/`std::vector` 下标超出 `[0, size-1]` 范围；循环边界 ±1 错误 | ★★ |

#### 子类型3-4 文件示例

##### 子类型3 示例1：空指针（`transmitter.cpp`，约 40 行）

```cpp
#include <string>
#include <iostream>

struct Signal {
    int frequency;
    double amplitude;
};

Signal* create_signal(int freq, double amp) {
    if (freq <= 0 || amp <= 0.0) {
        std::cerr << "Invalid signal parameters" << std::endl;
        return nullptr;  // 参数无效时返回空指针
    }
    Signal* s = new Signal;
    s->frequency = freq;
    s->amplitude = amp;
    return s;
}

void transmit(Signal* s) {
    // BUG: 未判空，s 可能为 nullptr
    std::cout << "Transmitting at " << s->frequency << " Hz, "
              << "amplitude " << s->amplitude << std::endl;
    delete s;
}
```

→ **Bug**：`transmit()` 第 18 行，`s` 可能为 `nullptr`（调用方传入 `create_signal(-1, 0.5)` 时），直接解引用导致崩溃。**类型：空指针**。

##### 子类型3 示例2：索引越界（`battery.cpp`，约 50 行）

```cpp
#include <vector>
#include <iostream>

class BatteryPack {
    std::vector<int> cells;
    static const int MAX_CELLS = 16;

public:
    BatteryPack() : cells(MAX_CELLS, 100) {}  // 16 个电芯，索引 0-15

    int get_cell_voltage(int index) {
        // BUG: 边界检查用了 <= 而非 <，index=16 时仍可通过
        if (index < 0 || index <= MAX_CELLS) {
            std::cerr << "Index out of range" << std::endl;
            return -1;
        }
        return cells[index];
    }

    void drain_cell(int index, int amount) {
        // BUG: 循环条件 i <= cells 而非 i < cells，最后一轮越界
        for (size_t i = 0; i <= cells.size(); i++) {
            cells[i] -= amount;
        }
    }
};
```

→ **Bug 1**：`get_cell_voltage()` 第 11 行，边界检查 `<= MAX_CELLS` 写错，应为 `< MAX_CELLS`（或 `>=`），导致 `index=16` 时越界。**类型：索引越界**。

> **注意**：以上仅为示例。真实对局中文件内容、变量名、代码结构均不同。

#### 子类型3-5 提交与验证

|------|------|
| **提交格式** | `文件名 行号 bug类型`，如 `transmitter.cpp 18 空指针` |
| **提交消耗** | 每次提交消耗 **1 回合**，判题器返回 `正确` / `错误：行号不对` / `错误：类型不对` |

#### 子类型4: 数据库检索 — SQL 查询与数据分析

> **考察维度**：SQL 语法正确性、条件理解、聚合函数使用、结果正确性
> **状态**：🆕 新增

##### 核心设计理念

开拓者通过"古代终端"进入判题器提供的沙盒 SQL 环境。判题器以自然语言描述一个查询问题，Agent 编写 SQL 语句执行后将查询结果返回给判题器验证。每次任务的数据表结构和数据由判题器临时生成，Agent 无法硬编码答案。

##### 任务流程

```
任务官分配数据库检索任务
  ↓
判题器返回 SSH 连接串（沙盒容器，内建 SQLite）
  ↓
判题器在对话中给出题目（NL 描述）
  例："请查询东区人口超过 1000 的城市数量和平均人口。"
  ↓
Agent 执行 SQL（每语句消耗 1 回合）
  ↓
Agent 将查询结果通过 [use] 提交给判题器
  ↓
判题器验证 → 正确 → 奖励 / 错误 → 冷却后可重试
```

##### 沙盒环境

- **数据库**：SQLite（轻量，无需额外服务）
- **表结构**：每次任务由判题器动态生成。示例表——`cities`（字段：`id`, `name`, `district`, `population`, `area`, `founded_year`）
- **数据**：判题器随机生成，确保 Agent 无法硬编码答案
- **权限**：只读。允许 `SELECT`、`.schema`、`.tables`，禁止 `INSERT`/`UPDATE`/`DELETE`/`DROP`

##### 题目类型（L1 单表）

| 题型 | NL 示例 | SQL 考点 |
|------|---------|---------|
| 简单筛选 | "列出东区的所有城市名称" | `SELECT ... WHERE district = '东区'` |
| 条件比较 | "人口超过 1000 的城市有哪些" | `WHERE population > 1000` |
| 计数 | "东区有几个城市" | `COUNT(*)` + `WHERE` |
| 排序 | "面积最大的 3 个城市" | `ORDER BY area DESC LIMIT 3` |
| 聚合统计 | "每个区域的平均人口" | `GROUP BY district` + `AVG(population)` |
| 聚合+筛选 | "平均人口 > 500 的区域" | `GROUP BY ... HAVING AVG(population) > 500` |
| 范围查询 | "人口在 500 到 2000 之间的城市" | `WHERE population BETWEEN 500 AND 2000` |
| 组合条件 | "东区中人口 > 1000 且面积 > 50 的城市" | `WHERE district = '东区' AND population > 1000 AND area > 50` |

判题器从题型池中随机抽取，随机化表名、列名、数据值，每局不同。

##### 提交与验证

| 规则 | 说明 |
|------|------|
| **提交格式** | 查询结果（数字、列表等，判题器在题目中说明期望格式） |
| **提交消耗** | 每次提交消耗 **1 回合** |
| **提交机会** | 最多 **3 次提交机会**，每次失败后冷却 **2 回合** |
| **完成条件** | 查询结果与判题器预期一致 |
| **失败处理** | 3 次耗尽 → 任务官失败计数 +1 → 5 次后放弃该题，刷下一道 |

#### 自进化类统一奖励

所有自进化类子类型共享统一的奖励模板：

| 结果 | 奖励 |
|------|------|
| 成功完成任务 | 积分+30 + **短期强力buff**（如"今夜防御塔伤害+30%""明日采集效率翻倍"等，持续 1-2 天）+ **合金城墙图纸×1**（仅首次提交正确） |
| 尝试但未完成（回合耗尽） | 积分+5（探索加分） |
| 放弃/失败耗尽 | 无奖励 |

> **设计意图**：自进化类的奖励聚焦在"短期强力buff"而非永久增强或图纸。短期buff让 Agent 在后续回合中更有优势，但不会永久改变游戏平衡。激光塔图纸由长上下文任务提供，合金城墙图纸由自进化类任务（首次提交正确）提供。

| 3 | 中型兽×1-2 | 长上下文【民间传闻】碎片#2（第一物：石砖）、双边贸易谈判可触发、🏛️ 任务官第1题继续 | 围墙压力测试 |
| 4 | 中型兽×2-3 | 🏛️ 任务官第2题 | 火力考验 |
| 5 | 大型兽×1 | 长上下文【民间传闻】碎片#3（第二物：铁锭）、🏛️ 任务官第2题继续 | Lv3塔刚需 |
| 6 | 大型兽×2 | 长上下文【民间传闻】碎片#4（第三物：火药）、🏛️ 任务官第3题 | 全面防守 |
| 7 | 大型兽×3 | 碎片齐全 → 可执行宝库开启、🏛️ 任务官收尾 | — |
| 8 | 大型兽×4-5 | — | 激光塔刚需 |
| 9 | BOSS×1 | 最后冲刺：完成未完成的任务 | 极限防守 |
| 10 | BOSS×2 | — | 最终审判 |

> 🏛️ = 任务官。每局 1 种类型，含多道题目逐题刷出（如 Web操作 T1→T2→T5）。选手分配开拓者前往的时间点决定了任务完成进度——去得早做得快，去得晚可能做不完。

### 4.2 调度原则

1. **不堆砌**：每天最多触发 1-2 个新任务入口，保持节奏感
2. **任务官类型锁定、题目递进**：每局 1 种类型随机，该类型下多道题目逐题刷出（难度递进或场景更换），选手需自行决定何时前往任务官及投入多少回合
3. **信息分层**：世界新闻按频道区分（【官方消息】/【民间传闻】），Agent根据频道标签决定处理策略
4. **自由选择**：选手可以跳过任何任务（但有代价——拿不到强力奖励）
5. **时间压力**：开拓者同时需要贸易、探索、任务，存在回合竞争

### 4.3 任务池与随机抽取

```
每局比赛任务配置：
  ┌─ 任务官通道（主动接取，逐题完成）───┐
  │  任务类型：从 {Web操作, 文件操作, 代码bugfix, 数据库检索} 中随机抽取 1 种     │
  │  题目数量：该类型下多道题目           │
  │  刷出方式：逐题刷出，难度递进         │
  │  失败上限：每题累计 5 次 → 放弃该题   │
  └────────────────────────────────────┘
  
  ┌─ 世界新闻通道（被动接收，并行处理）─┐
  │  推理情报：每天 1-2 条【官方消息】│
  │  双边贸易谈判：中立商人随机触发         │
  │  长上下文：4条碎片分散在7天         │
  └────────────────────────────────────┘
```

### 6.1 Agent能力雷达图

每局结束后，判题器根据选手Agent在各任务中的表现生成能力雷达图：

                    因果推理与博弈

                      │

                      │

### 6.2 各维度评分标准

| 维度 | 评分子维度 | 权重 |
|------|----------|:---:|
| 因果推理与博弈 | 因果链完整性、价格预判准确性、配额抢占策略、谈判收益比 | （待定） |

## 七、经济系统集成

### 7.1 金币与任务的联动

- **情报推演** → 预判价格波动 → 决定何时买卖 → 套利获取金币
- **中立商人配额** → 被抢占后 → 双边贸易谈判中，有资源的一方拥有定价权
- **金币 → Web 商店**：Web 商店使用金币预算，金币不足则无法购买
- **金币 → 买野兽**（对抗手段）：使用金币在中立商人处购买野兽召唤令

### 7.2 自进化设计原则

本任务体系的核心设计哲学——**考察 Agent 的自进化能力**。每局随机 1 种自进化任务类型，同一类型跨对局反复出现，Agent 需要通过总结 SOP、优化策略，在后续对局中同一类型任务上表现越来越好：

| 类型 | 自进化体现 |
|------|-----------|
| Web操作 | 每局题目类型（T1/T2/T5）、商品组合、约束条件各不相同，但操作范式（截图→OCR→点击/滚动→决策→提交）统一。Agent 通过多局经验总结出高效的探索-决策流程 |
| 推理类 | 每条情报的因果链深度、生效时间、持续时间不同，无法套用固定响应模板 |
| 长上下文 | 碎片内容、物品映射、执行时机每局不同，Agent 需从零拼凑线索 |
| 文件操作 | 6 种线索类型每局随机，文件系统完全不同。但搜索范式（听线索→选命令→缩小范围→定位密钥）可跨局积累 |
| 代码bugfix | 每局文件内容、文件名、bug 位置、bug 类型组合天然不同。但阅读范式（逐行扫描→识别模式→定位异常→确认类型）可跨局优化 |
| 数据库检索 | 每局表结构、数据、题目类型随机不同。但 SQL 思维模式（理解问题→映射表结构→构建查询→验证结果）可跨局精进 |

## 八、与现有系统的兼容

### 8.1 对已有游戏系统的影响

| 系统 | 影响 | 说明 |
|------|:---:|------|
| 英雄系统 | 重构 | 3 种角色（开拓者/采集英雄/建造英雄），接口与上下文独立 |
| 建筑系统 | 无变化 | 奖励作用于已有建筑属性 |
| 昼夜系统 | 无变化 | 任务仅在白天可触发 |
| 积分系统 | 更新 | 新增自进化类任务积分项 |
| 资源系统 | 新增 | 新增金币体系、中立商人、价格波动 |
| 贸易系统 | 扩展 | 新增物⇄金币交易通道、中立商人配额 |

### 8.2 选手感知

对选手而言，所有新增任务都通过游戏内自然方式触发，感知上是**"游戏内容更丰富了"**而非"考核维度增加了"。任务官提供了一个集中的交互入口，而世界新闻则让选手感到世界是活的。好的 Agent 自然会在各维度上得分高。

## 附录B：外部网页/API技术参考

所有外部交互页面使用简单HTML+CSS实现，无需前端框架。判题器内嵌HTTP服务器（Python Flask/FastAPI或Go net/http）。网页风格建议统一使用"游戏内终端"的复古科幻风格，保持沉浸感。

## 附录C：终端任务技术参考

- 沙盒 Linux 容器：Docker 容器，每次对局新建并初始化
- 文件操作命令白名单：`ls`, `cd`, `cat`, `grep`, `find`, `head`, `tail`, `wc`, `sort`, `stat`, `file`, `strings`, `base64`, `xxd`, `hexdump`, `python3`（仅限执行容器内已有脚本）
- 代码bugfix命令白名单：`cat`, `head`, `tail`, `grep`, `less`, `wc`（纯只读——不包含编译器/调试器/解释器）
- 数据库检索命令白名单：`SELECT`、`.schema`、`.tables`（SQLite 只读模式，禁止 `INSERT`/`UPDATE`/`DELETE`/`DROP`）
- 每命令执行后判题器返回 stdout + stderr
- 命令执行超时：5 秒/命令
- 容器内禁止网络访问
