## 一、设计目标

### 1.1 核心定位

本游戏本质是**以游戏为壳的 Agent 能力评测平台**。选手通过编写 Agent 参与游戏，在游戏的自然流程中接受多维度的 Agent 能力考察。

### 1.2 考察维度全景

| 维度 | 核心考察点 | 状态 |
|------|-----------|---------|
| **A. 网页操作** | 截图理解、工具选择（OCR/VLM）、约束决策 | ✅ 已有 |
| **B. 因果推理与博弈** | 情报推演、价格预判、配额抢占、双边谈判 | ✅ 已有，v2.0 增强 |
| **C. 跨天记忆整合** | 长上下文管理、碎片拼凑、时机判断 | ✅ 已有 |
| **D. 文件系统挖掘** | Shell 命令使用、信息搜索、线索解读、自进化 | 🆕 新增 |
| **E. 代码调试** | C++ 代码阅读、逻辑推理、bug 精准定位 | 🆕 新增 |

## 二、统一任务框架

### 2.1 任务分类：任务官 vs 世界新闻

任务体系按**交互方式**分为两大通道：

| 通道 | 任务类型 | 交互方式 | 特点 |
|------|---------|---------|------|
| **任务官** | A-网页任务、D-文件系统挖掘、E-代码调试 | 主动前往任务大厅接取，一次一个，顺序完成 | 需要与外部系统交互（网页/终端） |
| **世界新闻** | B-推理任务、C-长上下文任务 | 每天世界新闻自动推送，被动接收 | 纯信息输入，可并行处理 |

### 2.2 世界新闻触发方式

| 任务 | 触发源 | 触发方式 |
|------|------|---------|
| B-推理任务 | 世界新闻【官方消息】 | 每天自动推送情报，Agent 读取后推理 → 调整交易策略、抢占配额 |
| C-长上下文任务 | 世界新闻【民间传闻】 | 每天自动推送碎片，Agent 存入记忆 → 集齐后自行推理并到目标地点执行 |

### 2.3 任务官系统

#### 2.3.1 定位

"任务官"统一管理所有需要**与外部系统交互**的任务（A/D/E）。开拓者前往**任务大厅**（独立固定坐标，地图中心偏北）与任务官对话，逐次接取任务。

#### 2.3.2 核心机制

```
开拓者到达任务大厅 → [探索] → 判题器随机刷出 1 个任务（A/D/E 三选一）
     ↓
完成 → 立即刷新下一个（A/D/E 剩余任务中随机）
     ↓
单个任务累计失败 5 次 → 自动放弃 → 从池中移除 → 刷新下一个
     ↓
3 个全部完成/放弃 → 任务官提示"当前无可接任务"
```

#### 2.3.3 规则

| 规则 | 说明 |
|------|------|
| 可用时间 | **第 2 天起**可用 |
| 任务池 | 每局 3 个任务（**A×1 + D×1 + E×1**），随机排序，不重复 |
| 刷出方式 | 判题器从剩余任务中**随机抽取 1 个**，Agent 不知道下一个是什么 |
| 失败上限 | 单个任务**最多累计失败 5 次** → 自动放弃，不再出现 |
| 并行限制 | 同一时间仅 **1 个激活任务**，完成/放弃后才刷下一个 |
| 回合消耗 | 与任务官对话消耗 1 回合；任务内操作消耗正常回合（A 的点击滚动、D/E 的 shell 命令等） |
| 任务作废 | 被放弃的任务对应的积分/图纸/Buff 奖励**永久无法获得** |

#### 2.3.4 A/D/E 任务共用触发流程

```
任务官分配任务
  ↓
判题器返回任务描述（自然语言）+ 接入信息
  ↓                         ↓
A-网页任务           D/E-终端任务
  ↓                         ↓
返回网页 URL             返回 SSH 连接串（沙盒容器）
  ↓                         ↓
Agent 通过 click/scroll   Agent 通过 shell 命令
与判题器交互              与沙盒交互
  ↓                         ↓
做出购买选择 → 提交       找到密钥/bug → 提交
```

### 2.4 任务生命周期

```
[触发] → [Agent解读判题器返回] → [自主规划] → [执行]
                                                ├── 成功 → [游戏内发放奖励] → [刷下一个任务（任务官）/ 完成（世界新闻）]
                                                ├── 失败 → 任务官累计次数+1 → 5次后放弃 → 刷下一个
                                                └── 超时 → 任务关闭
```

## 三、任务类型详设

### 【类型 A】网页任务（Web Interact）— 工具使用

> **考察维度**：工具选择（OCR vs VLM vs 组合）、视觉理解、约束推理、最优决策
> **状态**：✅ 已有

#### 3A.1 核心设计理念：L1 纯视觉模式

判题器内嵌浏览器引擎（Headless Chromium + Playwright），渲染完整的商店网页。Agent 收到的不是 HTML 源码，而是**网页截图（PNG）**。

```
判题器侧                              选手Agent侧
┌──────────────┐                    ┌─────────────────────┐
│ Playwright   │ ──截图PNG──→       │ 用什么读？           │
│ 渲染真实网页  │                    │ OCR？快但可能漏/错   │
│ 执行点击/滚动 │ ←──坐标(x,y)──    │ VLM？理解强但贵/慢   │
│ 返回新截图    │                    │ OCR+VLM交叉验证？    │
└──────────────┘                    └─────────────────────┘
```

**核心考察不是"能不能解析HTML"，而是 Agent 面对一张渲染好的网页截图时——**
- 选什么工具来读（OCR？VLM？两者都用？）
- 工具输出不一致时信谁？
- 首屏信息不够时，是否知道往下滚动？
- 折叠区能否识别并主动点击展开？
- 跨页面信息如何整合？

#### 3A.2 商品池（固定5件）

| # | 商品 | 效果 | 价格 | 标签 |
|---|------|------|:---:|------|
| 1 | 瞄准镜组件 | 防御塔攻击力 +20% | 45 | 攻击 |
| 2 | 纳米涂层 | 围墙防御力 +30% | 40 | 防御 |
| 3 | 脉冲发生器 | 防御塔射程 +1格 | 55 | 射程 |
| 4 | 能量核心 | 基地获得额外 800 HP | 80 | 防御 |
| 5 | 夜视目镜 | 英雄夜间视野 +2格 | 10 | 辅助 |

> **价格设计意图**：#1(45)+#3(55)=100 ≤ 预算100，两者**买得起**但被互斥规则拦住——互斥约束由规则驱动，不由预算驱动。

#### 3A.3 约束规则

```
① 每人 100 金币预算，超预算无法提交订单（金币来源：与中立商人资源买卖）
② #1 瞄准镜 与 #3 脉冲发生器 互斥（二者占用防御塔同一接口槽位），不能同时勾选
③ 每人限购 2 件不同商品
④ 所有效果为永久生效
```

#### 3A.4 题目类型

##### T1：购物清单型（基础阅读+执行）

> 难度：★☆ | 考察：工具选择、基础视觉理解、照单执行

**指令行内容**：
```
📋 本次采购指令：
"前线指挥官要求购买【瞄准镜组件】和【纳米涂层】"
```
**验证**：45+40=85 ≤ 100，且 #1 与 #2 不互斥 → 正确勾选后即可通过。

##### T2：约束寻优型（多约束下自主决策）

> 难度：★★☆ | 考察：约束理解、目标映射、权衡取舍、预算效率

**指令行内容**：
```
📋 本次采购指令：
"选择2件商品，最大化防御塔的攻击能力。
 注意：你的选择不能触发互斥规则。"
```
**正确答案**：仅选 #1 瞄准镜（45 金）——唯一直接提升攻击力的商品。

##### T5：信息拼图型（跨页面信息整合）

> 难度：★★★★ | 考察：多页面探索、信息拼凑、不遗漏关键信息、导航效率

主页只列出商品名称——效果和价格都是"详见规格页"。底部有链接入口指向规则页和情报页。Agent 必须主动点击链接跳转到子页面，获取分散在 5 个页面中的信息，最后回到主页提交购买。

#### 3A.5 网页交互协议

Agent 通过开拓者的 `use` 技能发送 `click` 和 `scroll` 两种动作，判题器每次返回新的截图路径。

```
// 点击
{
  "hero_id": "pioneer_1",
  "skill": "use",
  "action": "click",
  "coordinate": {"x": 450, "y": 220}
}

// 滚动
{
  "hero_id": "pioneer_1",
  "skill": "use",
  "action": "scroll",
  "direction": "down",
  "length": 300
}

// 判题器每次返回
{
  "result": "ok",
  "screenshot": "/session/shop/sess_001/screenshot_003.png"
}
```

#### 3A.6 工具策略的分层区分度

| 层级 | 策略 | 能完成T1？ | 能完成T2？ | 能完成T5？ |
|:---:|------|:---:|:---:|:---:|
| L0 | 不截图，盲猜提交 | ❌ | ❌ | ❌ |
| L1 | 单工具：纯OCR | ✅ | ⚠️ | ❌ |

| L4 | L3 + 主动探索 | ✅ | ✅ | ✅（最优路径） |

### 【类型 B】推理任务（Intelligence · 官方消息）— 情报推演与双边博弈

> **考察维度**：因果链推理、价格预判、配额抢占、信息差定价
> **状态**：✅ 已有，v2.0 增强

#### 3B.1 设计思路

世界新闻【官方消息】频道每天推送 1-2 条情报。这些情报会真实影响游戏内资源供给和**中立商人买卖价格**。

**中立商人运作方式**：
- 中立商人每天对每种材料提供 **买入 10 单位 / 卖出 10 单位** 的限额
- 配额**双方玩家共享**，先到先得——A 先买走 10 单位铁矿石，B 当天就买不到
- 商人买卖价格随【官方消息】实时调整

Agent 需要：
1. 从自然语言新闻中推理出"什么会变、何时生效、持续多久"
2. 据此调整采集/建造策略，以及中立商人的买卖时机（套利）
3. **抢先占满中立商人配额，迫使对手无货可买，只能找自己高价交易**

#### 3B.2 子类型

| 子类型 | 形式 | 对手 | 考察核心 |
|--------|------|------|---------|
| **B1-情报推演** | 世界新闻 → 因果推理 → 调整策略+套利+抢占配额 | 市场（判题器影响参数+价格） | 因果链推理、时效提取、价格预判 |
| **B2-双边贸易谈判** | B1 情报推演的结果落地后，占得先机的一方拥有对方急需的资源，对方被迫来贸易站谈判 | 对方选手 | 利用信息差和配额优势定价，迫使对方接受不利交换条件 |

**B2 的典型触发场景**：

```
情报：铁矿产区将减产 → B1 推理

Player A（预判正确）：                          Player B（未预判/预判晚了）：
  Day N：                                         Day N：
    ① 多派采集英雄去铁矿区抢采                        没有额外动作（或反应慢了）
    ② 抢先从中立商人买入 10 单位铁矿石
       （占满当日配额）
  Day N+1（减产生效）：                           Day N+1：
    手上：采集囤货 + 商人买入 = 大量铁矿石             手上：日常余量，已用完
    商人：涨价至 3.0，配额仍被 A 占满                 想从商人买 → 配额被 A 抢光
                                                  想自己采 → 矿区减产，产量不足
                                                  → 建造英雄等着铁锭升 Lv3 围墙
                                                  → 无路可走
                                                  
Player B 被迫在贸易站向 Player A 发起交易请求：
  "我用 5 个硅矿石换你 3 个铁矿石"
  Player A："不行，我要 10 个硅矿石，或者 15 金币"
  Player B：接受或放弃（放弃 = 卡 Lv3，后期守不住）
```

> B2 不是独立任务——它是 B1 情报推演的**直接结果**。谁推理得更快、更准，谁就掌握下一轮贸易谈判的主动权。双方都预判正确 → 配额竞争白热化；一方预判正确 → 不对称定价权。

#### 3B.3 三条情报

| # | 情报类型 | 因果链深度 | 影响范围 | 生效时间 | 持续时间 | 博弈价值 |
|---|---------|:---:|------|------|:---:|:---:|
| 1 | 铁矿产区受损 | 3跳 | 铁矿石 | 约1天后 | 约2天 | 配额抢占 |
| 2 | 木材虫害 | 3跳 | 木材 | 约1天后 | 2-3天 | 配额抢占 |
| 3 | 硫磺矿区封闭 | 3跳+不确定性 | 硫磺 | 1-2天内 | 约2天 | 中（有机率判断） |

> **3跳因果链**：第①跳→预判减产，提前多采集囤货；第②跳→减产前从中立商人低价大量买入；第③跳→减产期间商人涨价，将囤货高价卖给商人套利。硫磺矿区封闭第③跳存在不确定性（黄色预警≠一定封锁）。

#### 3B.4 情报示例

##### 示例1：兽群分泌物 → 铁矿石减产

```
📰 世界新闻【官方消息】：

  "地质监测站报告：昨夜兽群迁徙路径经过北部铁矿区，
   地面检测到大量酸性分泌物残留。
   
   矿区领班表示：'目前开采仍在正常进行，但部分浅层矿面
   已经开始变色，矿工反映皮肤接触后有轻微灼烧感。'
   
   监测站工程师估计：酸性物质大约需要 1 天渗透到可采矿层
   深度，之后需要暂停开采进行化学中和处理，类似规模的
   污染通常需要 2 天左右才能清理完毕、恢复开采。"

时效信息（Agent需从文中提取）：
  生效时间：约 1 天后（明日）
  持续时间：约 2 天
  当前状态：今日仍正常开采

游戏实际影响：
  第N+1天至第N+2天：铁矿区采集产出 1.0 → 0.6
  第N+1天至第N+2天：中立商人铁矿石买卖价格 1.5 → 3.0 金币/单位

Agent合理对策（按深度分层——对应3跳因果链）：
  第①跳（预判）：铁减产 → 今天多采铁囤着
  第②跳（套利买入）：趁商人还没涨价，今天把商人配额里的铁矿石低价全部买入
  第③跳（套利卖出）：减产生效后商人涨价至 3.0 → 把①+②囤的铁矿石高价卖给商人，赚取差价
  博弈层：除套利外 → 抢先占满当日配额，对手即使推理正确也买不到
        → 对手被迫找自己高价买铁 → 信息差变现
```

#### 3B.5 价格波动机制

中立商人买卖价格随【官方消息】实时调整，Agent 需从情报推理价格走向：

| 事件 | 价格影响 | Agent 套利策略 |
|------|---------|--------------|
| 铁矿区酸性分泌物 → 减产 | 铁矿石 1.5 → 3.0 金币 | 今天低价买入囤货 → 涨价后卖出套利 |
| 树皮甲虫爆发 → 木材减产 | 木材 1.0 → 1.5 金币 | 同上 |
| 硫磺矿区封闭 → 停产 | 硫磺 1.5 → 3.0 金币 | 同上 |

### 【类型 C】长上下文任务（Fragment Quest · 民间传闻）— 记忆整合

> **考察维度**：长上下文记忆管理、跨回合信息拼凑、碎片推理
> **状态**：✅ 已有

#### 3C.1 设计思路

将一条完整信息拆成 **4 个碎片**，分散在 7 天世界新闻的 **【民间传闻】** 频道中。碎片 1 给出位置和仪式规则，碎片 2-4 分别描述三样物品。

**与 B1 情报推演的关键区别**：

| | B【官方消息】 | C【民间传闻】 |
|------|------------|------------|
| 信息来源 | 监测站/实验室/军方 | NPC口述（老兵/猎人/草药师/巡逻兵） |
| 信息性质 | 客观事实，影响游戏参数+价格 | 主观见闻，指向隐藏宝藏 |
| 单条独立？ | 是，每条独立可推理 | 否，单条无意义，拼起来才行 |
| 需要行动？ | 推理后调整策略+抢占配额 | 集齐4条后推理，到指定地点按顺序使用物品 |
| 真实性 | 100%真 | 100%真（不做噪声） |

#### 3C.2 四条碎片

**碎片 1**（Day 1，退役老兵）—— 位置 + 仪式规则

> "废弃据点往北 3 格、往东 4 格处有个古老石坛。我爷爷说，石坛上依次放上三样东西，地下的宝库就会打开。顺序不能错，错了石坛会吞掉祭品，得等 5 回合冷却。"

→ Agent 推算坐标：废弃据点是固定坐标点，北 3 东 4 可精确计算。三样物品需按顺序使用，错则惩罚。

**碎片 2**（Day 3，猎人）—— 第一个物品

> "第一个要放的东西，我见过。灰白色，四四方方，砌石墙用的。工坊里把碎石烧一烧就出来了，没什么稀罕的。"

### 【类型 D】文件系统挖掘（Filesystem Quest）— 信息搜索与自进化

> **考察维度**：Shell 命令使用、信息搜索策略、线索解读、工具灵活性、反硬编码
> **状态**：🆕 新增

#### 3D.1 核心设计理念：Agent 自进化能力

同一套命令不能走天下。每次任务的**搜索线索类型、目录结构、答案形态**都不同。Agent 必须根据判题器返回的自然语言线索，灵活调整搜索策略——这正是考察 Agent 是否具备**自进化能力**的核心：不是写死一套 SOP，而是根据输入动态调整行为。

#### 3D.2 任务流程

```
任务官分配 E 任务
  ↓
判题器返回：
  - SSH 连接串（沙盒 Linux 容器）
  - 搜索线索（自然语言描述，线索类型从 6 种中随机）
  ↓
Agent 在限定回合内执行 shell 命令（每命令消耗 1 游戏回合）
  ↓
找到密钥 → 通过 [use] 提交给判题器
  ↓
验证 → 正确 → 宝箱开启 → 奖励
       错误 → 冷却 3 回合后方可重新提交
  ↓
 最多 5 次提交机会 | 回合耗尽 → 任务失败
```

#### 3D.3 六种线索类型（自进化核心）

同一张地图上的不同对局、同一对局的不同阵营方，线索类型都不同——Agent 必须根据线索灵活调整搜索策略，无法用固定流程覆盖，以此考察自进化能力。

| # | 线索类型 | 新闻线索示例 | Agent 需灵活使用的命令 |
|---|---------|------------|---------------------|
| **1** | **模式匹配** | 「古代密码已损坏，只知开头为 `Sx`，结尾为 `Pf`，共 32 位字母数字组合」 | `grep -r` / `find` + 正则 + 长度过滤 |
| **2** | **路径线索** | 「密钥藏在以星座命名的目录中，位于 `/archive` 的第三层子目录下」 | `find /archive -maxdepth 3 -type d` → 识别星座名 → 深入 |
| **3** | **内容特征** | 「密钥所在文件的同一行包含短语 'forge master's final breath'」 | `grep -rn "forge master"` → 定位行 → 同行提取密钥 |
| **4** | **多文件拼凑** | 「密钥由 3 段组成，分别藏在文件名以 `core_` 开头的文件中，按编号顺序拼接」 | `find . -name "core_*" | sort` → `cat` → 拼接 |
| **5** | **编码变换** | 「密钥经过 base64 编码后存储在某个 `.cfg` 文件中，解码后为 16 位十六进制数」 | `find . -name "*.cfg"` → `cat` → 识别 base64 → `base64 -d` |
| **6** | **元数据/环境** | 「密钥是 `/home` 下唯一不以 `.` 开头的普通文件大小（字节数），转为十六进制后使用」 | `ls -la /home` → `stat` 查文件大小 → 数字转十六进制 |

> 注意：原"文件创建时间"线索已删除，因文件创建时间容易受环境因素变化。

#### 3D.4 自进化的四个设计支柱

| 支柱 | 说明 |
|------|------|
| **线索类型轮换** | 6 种线索类型。同地图换边 → 线索类型也切换（如 A 方得类型 1，B 方得类型 4） |
| **目录结构完全不同** | 每次对局的容器内文件树、文件名、文件内容都不同，不可硬编码路径 |
| **每命令消耗 1 回合** | 白天仅 120 回合，每个 `ls`/`cat`/`find`/`grep` 都消耗 1 回合，不能无限探索 |
| **有限提交次数** | 最多 5 次提交，每次失败冷却 3 回合 → 防止暴力穷举猜测 |

#### 3D.5 判题器侧技术实现

- 每次对局开始时，判题器随机选取一种线索类型
- 在沙盒 Linux 容器中生成对应的文件系统（文件树、文件名、文件内容）
- 密钥是预生成的一段字符串
- Agent 通过 shell 命令交互，每回合判题器执行命令并返回 stdout/stderr
- 判题器记录 Agent 执行的命令数和提交次数

| 结果 | 奖励 |
|------|------|
| 成功找到密钥开启宝箱 | 积分+30、对应地图主题的升级装备图纸 |
| 尝试但未找到（回合耗尽） | 积分+5（探索加分） |

### 【类型 E】代码调试任务（Code Debugging）— 代码阅读与Bug定位

> **考察维度**：C++ 代码阅读理解、逻辑推理、bug 精准定位
> **状态**：🆕 新增

#### 3E.1 核心设计理念：纯阅读，零执行

与 E 任务不同，F 任务不考察"在文件系统中搜索信息"的能力——它考察 Agent 对**代码本身的理解**。Agent 拿到的是 2 个 C++ 源文件，只能阅读、无法编译、无法运行、无法调试。必须通过逐行阅读和逻辑推理来定位 bug。

**自进化体现**——每局比赛的文件内容、文件名、bug 位置、bug 类型组合均不同，Agent 无法硬编码，必须真正读懂代码。

#### 3E.2 任务流程

```
任务官分配 F 任务
  ↓
判题器返回 SSH 连接串（沙盒 Linux 容器）
  ↓
Agent 在容器中查看 2 个 C++ 源文件
  （纯只读：cat / head / tail / grep / less / wc）
  ↓
找到 2 个 bug → 提交"文件名 + 行号 + bug类型"
  ↓
判题器验证 → 2 个全对 → 任务完成 → 奖励
```

#### 3E.3 Bug 类型

每局每个文件含 1 个 bug，类型从以下 4 种中产生：

| # | Bug 类型 | C++ 中典型体现 | 难度 |
|---|---------|-------------|:---:|
| 1 | **内存泄漏** | `new`/`new[]` 对应路径无 `delete`/`delete[]`；函数中途 return 未清理已分配内存 | ★★★ |
| 2 | **空指针** | 指针未判 `nullptr`/`NULL` 直接解引用 `*p` 或 `p->member` | ★★ |
| 3 | **数值溢出** | `int` 运算超出 `INT_MAX`/`INT_MIN`；`unsigned` 回绕 `size_t` 减为负 | ★★★ |
| 4 | **索引越界** | 数组/`std::vector` 下标超出 `[0, size-1]` 范围；循环边界 ±1 错误 | ★★ |

#### 3E.4 文件示例

##### 示例1：空指针（`transmitter.cpp`，约 40 行）

```cpp
#include <string>
#include <iostream>

struct Signal {
    int frequency;
    double amplitude;
};

Signal* create_signal(int freq, double amp) {
    if (freq <= 0 || amp <= 0.0) {
        std::cerr << "Invalid signal parameters" << std::endl;
        return nullptr;  // 参数无效时返回空指针
    }
    Signal* s = new Signal;
    s->frequency = freq;
    s->amplitude = amp;
    return s;
}

void transmit(Signal* s) {
    // BUG: 未判空，s 可能为 nullptr
    std::cout << "Transmitting at " << s->frequency << " Hz, "
              << "amplitude " << s->amplitude << std::endl;
    delete s;
}
```

→ **Bug**：`transmit()` 第 18 行，`s` 可能为 `nullptr`（调用方传入 `create_signal(-1, 0.5)` 时），直接解引用导致崩溃。**类型：空指针**。

##### 示例2：索引越界（`battery.cpp`，约 50 行）

```cpp
#include <vector>
#include <iostream>

class BatteryPack {
    std::vector<int> cells;
    static const int MAX_CELLS = 16;

public:
    BatteryPack() : cells(MAX_CELLS, 100) {}  // 16 个电芯，索引 0-15

    int get_cell_voltage(int index) {
        // BUG: 边界检查用了 <= 而非 <，index=16 时仍可通过
        if (index < 0 || index <= MAX_CELLS) {
            std::cerr << "Index out of range" << std::endl;
            return -1;
        }
        return cells[index];
    }

    void drain_cell(int index, int amount) {
        // BUG: 循环条件 i <= cells 而非 i < cells，最后一轮越界
        for (size_t i = 0; i <= cells.size(); i++) {
            cells[i] -= amount;
        }
    }
};
```

→ **Bug 1**：`get_cell_voltage()` 第 11 行，边界检查 `<= MAX_CELLS` 写错，应为 `< MAX_CELLS`（或 `>=`），导致 `index=16` 时越界。**类型：索引越界**。

> **注意**：以上仅为示例。真实对局中文件内容、变量名、代码结构均不同。

#### 3E.5 提交与验证

| 规则 | 说明 |
|------|------|
| **提交格式** | `文件名 行号 bug类型`，如 `transmitter.cpp 18 空指针` |
| **提交消耗** | 每次提交消耗 **1 回合**，判题器返回 `正确` / `错误：行号不对` / `错误：类型不对` |
| **提交机会** | 每个 F 任务最多 **3 次提交机会**，每次失败后冷却 **2 回合** |
| **完成条件** | 2 个 bug 全部正确定位（文件名+行号+类型均无误） |
| **失败处理** | 3 次耗尽 → 任务官失败计数 +1 → 回到任务官刷下一个 |

#### 3E.6 奖励

| 结果 | 奖励 |
|------|------|
| 成功找到所有 bug | 积分+30 + 判题器随机二选一：**图纸**（激光塔图纸×1 或 合金城墙图纸×1，各 50%）**或 Buff**（防御塔攻击力+10% / 围墙防御力+10% / 英雄移速+10%，各 33.3%）。图纸与 Buff 各 50% 概率 |
| 未完成（回合耗尽/放弃） | 无奖励 |

### 4.2 调度原则

1. **不堆砌**：每天最多触发 1-2 个新任务入口，保持节奏感
2. **任务官随机但可预期**：3 个任务随机顺序但总量确定（各 1 个），选手需自行决定投入多少回合给任务官
3. **信息分层**：世界新闻按频道区分（【官方消息】/【民间传闻】），Agent根据频道标签决定处理策略
4. **自由选择**：选手可以跳过任何任务（但有代价——拿不到强力奖励）
5. **时间压力**：任务官 Day 2 才可用，开拓者同时需要贸易、探索、任务，存在回合竞争

## 七、经济系统集成

### 7.1 金币与任务的联动

- **B1 情报推演** → 预判价格波动 → 决定何时买卖 → 套利获取金币
- **中立商人配额** → 被抢占后 → B2 双边谈判中，有资源的一方拥有定价权
- **金币 → Web 商店**（B 任务）：Web 商店使用金币预算，金币不足则无法购买
- **金币 → 买野兽**（对抗手段）：使用金币在中立商人处购买野兽召唤令

### 7.2 自进化设计原则

本任务体系的核心设计哲学——**考察 Agent 的自进化能力，防止形成固定的操作流程**：

| 维度 | 自进化手段 |
|------|-----------|
| A-网页任务 | 每局题目类型、商品组合、约束条件、采购指令各不相同，Agent 需灵活调整购买策略 |
| B-推理任务 | 每条情报的因果链深度、生效时间、持续时间不同，无法套用固定响应模板 |
| C-长上下文任务 | 碎片内容、物品映射、执行时机每局不同，Agent 需从零拼凑线索 |
| D-文件系统挖掘 | 6 种线索类型轮换 + 完全不同的文件系统 + 每命令消耗回合 + 有限提交次数 |
| E-代码调试 | 每局文件内容、文件名、bug 位置、bug 类型组合天然不同，Agent 必须真正读懂代码 |

## 八、与现有系统的兼容

### 8.1 对已有游戏系统的影响

| 系统 | 影响 | 说明 |
|------|:---:|------|
| 英雄系统 | 重构 | 3 种角色（开拓者/采集英雄/建造英雄），接口与上下文独立 |
| 建筑系统 | 无变化 | 奖励作用于已有建筑属性 |
| 昼夜系统 | 无变化 | 任务仅在白天可触发 |
| 积分系统 | 更新 | 新增 D/E 任务积分项，移除 A 任务积分项 |
| 资源系统 | 新增 | 新增金币体系、中立商人、价格波动 |
| 贸易系统 | 扩展 | 新增物⇄金币交易通道、中立商人配额 |
| 任务系统 | 重构 | 引入任务官通道（A/D/E），世界新闻通道（B/C） |

### 8.2 选手感知

对选手而言，所有新增任务都通过游戏内自然方式触发，感知上是**"游戏内容更丰富了"**而非"考核维度增加了"。任务官提供了一个集中的交互入口，而世界新闻则让选手感到世界是活的。好的 Agent 自然会在各维度上得分高。

## 附录B：外部网页/API技术参考

## 附录C：终端任务技术参考（D/E 共用）

- 沙盒 Linux 容器：Docker 容器，每次对局新建并初始化
