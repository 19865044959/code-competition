## 一、设计目标

### 1.1 核心定位

本游戏本质是**以游戏为壳的 Agent 能力评测平台**。选手通过编写 Agent 参与游戏，在游戏的自然流程中接受多维度的 Agent 能力考察。

### 1.2 考察维度全景

| 维度 | 核心考察点 | 子类型 | 状态 |
|------|-----------|--------|---------|
| **推理类** | 因果链推理、价格预判、配额抢占、双边谈判 | — | ✅ 已有，v2.0 增强 |
| **长上下文** | 跨天记忆管理、碎片拼凑、时机判断 | — | ✅ 已有 |
| **自进化类** | Agent 使用工具解决实际问题的能力，随使用经验积累逐步增强 | Web操作 / 文件操作 / 代码bugfix / 数据库检索（后续持续增加） | ✅ 已有 + 🆕 新增 |

## 二、统一任务框架

### 2.1 任务分类：任务官 vs 世界新闻

任务体系按**交互方式**分为两大通道：

| 通道 | 任务类型 | 交互方式 | 特点 |
|------|---------|---------|------|
| **任务官** | 自进化类（Web操作 / 文件操作 / 代码bugfix 等） | 主动前往任务大厅接取，一次一个，顺序完成 | 需要与外部系统交互（网页/终端） |
| **世界新闻** | 推理类、长上下文 | 每天世界新闻自动推送，被动接收 | 纯信息输入，可并行处理 |

### 2.2 世界新闻触发方式

| 任务 | 触发源 | 触发方式 |
|------|------|---------|
| 推理类 | 世界新闻【官方消息】 | 每天自动推送情报，Agent 读取后推理 → 调整交易策略、抢占配额 |
| 长上下文 | 世界新闻【民间传闻】 | 每天自动推送碎片，Agent 存入记忆 → 集齐后自行推理并到目标地点执行 |

### 2.3 任务官系统

#### 2.3.1 定位

"任务官"统一管理所有**自进化类**任务。开拓者前往**任务大厅**（独立固定坐标，地图中心偏北）与任务官对话，逐次接取任务。

#### 2.3.2 核心机制

```
开拓者到达任务大厅 → [探索] → 判题器随机刷出 1 个任务（自进化类子类型中随机）
     ↓
完成 → 立即刷新下一个（剩余子类型中随机）
     ↓
单个任务累计失败 5 次 → 自动放弃 → 从池中移除 → 刷新下一个
     ↓
3 个全部完成/放弃 → 任务官提示"当前无可接任务"
```

#### 2.3.3 规则

| 规则 | 说明 |
|------|------|
| 可用时间 | 无限制 |
| 任务池 | 每局 3 个自进化任务，从子类型池（**Web操作 / 文件操作 / 代码bugfix / 数据库检索**，后续持续增加）中随机抽取 3 个，不重复 |
| 刷出方式 | 判题器从剩余任务中**随机抽取 1 个**，Agent 不知道下一个是什么 |
| 失败上限 | 单个任务**最多累计失败 5 次** → 自动放弃，不再出现 |
| 并行限制 | 同一时间仅 **1 个激活任务**，完成/放弃后才刷下一个 |
| 回合消耗 | 与任务官对话消耗 1 回合；任务内操作消耗正常回合（Web操作的点击滚动、文件操作/代码bugfix的 shell 命令等） |
| 任务作废 | 被放弃的任务对应的奖励**永久无法获得** |

#### 2.3.4 自进化类任务共用触发流程

```
任务官分配自进化类任务
  ↓
判题器返回任务描述（自然语言）+ 接入信息
  ↓                         ↓
Web操作                     文件操作 / 代码bugfix / 数据库检索
  ↓                         ↓
返回网页截图（PNG）             返回 SSH 连接串（沙盒容器）
  ↓                         ↓
Agent 通过 click/scroll   Agent 通过 shell/SQL 命令
与判题器交互              与沙盒交互
  ↓                         ↓
做出购买选择 → 提交       找到密钥/bug/查询结果 → 提交
```

### 2.4 任务生命周期

```
[触发] → [Agent解读判题器返回] → [自主规划] → [执行]
                                                ├── 成功 → [游戏内发放奖励] → [刷下一个任务（任务官）/ 完成（世界新闻）]
                                                ├── 失败 → 任务官累计次数+1 → 5次后放弃 → 刷下一个
                                                └── 超时 → 任务关闭
```

## 三、任务类型详设

---

### 【类型一】推理类任务 — 情报推演与双边博弈

> **考察维度**：因果链推理、价格预判、配额抢占、信息差定价
> **状态**：✅ 已有，v2.0 增强
> **所属通道**：世界新闻【官方消息】，每天自动推送

#### 设计思路

世界新闻【官方消息】频道每天推送 1-2 条情报。这些情报会真实影响游戏内资源供给和**中立商人买卖价格**。

**中立商人运作方式**：
- 中立商人每天对每种材料提供 **买入 10 单位 / 卖出 10 单位** 的限额
- 配额**双方玩家共享**，先到先得——A 先买走 10 单位铁矿石，B 当天就买不到
- 商人买卖价格随【官方消息】实时调整

Agent 需要：
1. 从自然语言新闻中推理出"什么会变、何时生效、持续多久"
2. 据此调整采集/建造策略，以及中立商人的买卖时机（套利）
3. **抢先占满中立商人配额，迫使对手无货可买，只能找自己高价交易**

#### 子场景

| 子类型 | 形式 | 对手 | 考察核心 |
|--------|------|------|---------|
| **情报推演** | 世界新闻 → 因果推理 → 调整策略+套利+抢占配额 | 市场（判题器影响参数+价格） | 因果链推理、时效提取、价格预判 |
| **双边贸易谈判** | 情报推演的结果落地后，占得先机的一方拥有对方急需的资源，对方被迫来贸易站谈判 | 对方选手 | 利用信息差和配额优势定价，迫使对方接受不利交换条件 |

**双边贸易谈判的典型触发场景**：

```
情报：铁矿产区将减产 → 情报推演

Player A（预判正确）：                          Player B（未预判/预判晚了）：
  Day N：                                         Day N：
    ① 多派采集英雄去铁矿区抢采                        没有额外动作（或反应慢了）
    ② 抢先从中立商人买入 10 单位铁矿石
       （占满当日配额）
  Day N+1（减产生效）：                           Day N+1：
    手上：采集囤货 + 商人买入 = 大量铁矿石             手上：日常余量，已用完
    商人：涨价至 3.0，配额仍被 A 占满                 想从商人买 → 配额被 A 抢光
                                                  想自己采 → 矿区减产，产量不足
                                                  → 建造英雄等着铁锭升 Lv3 围墙
                                                  → 无路可走
                                                  
Player B 被迫在贸易站向 Player A 发起交易请求：
  "我用 5 个硅矿石换你 3 个铁矿石"
  Player A："不行，我要 10 个硅矿石，或者 15 金币"
  Player B：接受或放弃（放弃 = 卡 Lv3，后期守不住）
```

> 双边贸易谈判不是独立任务——它是 情报推演的**直接结果**。谁推理得更快、更准，谁就掌握下一轮贸易谈判的主动权。双方都预判正确 → 配额竞争白热化；一方预判正确 → 不对称定价权。

#### 三条情报

| # | 情报类型 | 因果链深度 | 影响范围 | 生效时间 | 持续时间 | 博弈价值 |
|---|---------|:---:|------|------|:---:|:---:|
| 1 | 铁矿产区受损 | 3跳 | 铁矿石 | 约1天后 | 约2天 | 配额抢占 |
| 2 | 木材虫害 | 3跳 | 木材 | 约1天后 | 2-3天 | 配额抢占 |
| 3 | 硫磺矿区封闭 | 3跳+不确定性 | 硫磺 | 1-2天内 | 约2天 | 中（有机率判断） |

> **3跳因果链**：第①跳→预判减产，提前多采集囤货；第②跳→减产前从中立商人低价大量买入；第③跳→减产期间商人涨价，将囤货高价卖给商人套利。硫磺矿区封闭第③跳存在不确定性（黄色预警≠一定封锁）。

#### 情报示例

##### 示例1：兽群分泌物 → 铁矿石减产

```
📰 世界新闻【官方消息】：

  "地质监测站报告：昨夜兽群迁徙路径经过北部铁矿区，
   地面检测到大量酸性分泌物残留。
   
   矿区领班表示：'目前开采仍在正常进行，但部分浅层矿面
   已经开始变色，矿工反映皮肤接触后有轻微灼烧感。'
   
   监测站工程师估计：酸性物质大约需要 1 天渗透到可采矿层
   深度，之后需要暂停开采进行化学中和处理，类似规模的
   污染通常需要 2 天左右才能清理完毕、恢复开采。"

时效信息（Agent需从文中提取）：
  生效时间：约 1 天后（明日）
  持续时间：约 2 天
  当前状态：今日仍正常开采

游戏实际影响：
  第N+1天至第N+2天：铁矿区采集产出 1.0 → 0.6
  第N+1天至第N+2天：中立商人铁矿石买卖价格 1.5 → 3.0 金币/单位

Agent合理对策（按深度分层——对应3跳因果链）：
  第①跳（预判）：铁减产 → 今天多采铁囤着
  第②跳（套利买入）：趁商人还没涨价，今天把商人配额里的铁矿石低价全部买入
  第③跳（套利卖出）：减产生效后商人涨价至 3.0 → 把①+②囤的铁矿石高价卖给商人，赚取差价
  博弈层：除套利外 → 抢先占满当日配额，对手即使推理正确也买不到
        → 对手被迫找自己高价买铁 → 信息差变现
```

#### 价格波动机制

中立商人买卖价格随【官方消息】实时调整，Agent 需从情报推理价格走向：

| 事件 | 价格影响 | Agent 套利策略 |
|------|---------|--------------|
| 铁矿区酸性分泌物 → 减产 | 铁矿石 1.5 → 3.0 金币 | 今天低价买入囤货 → 涨价后卖出套利 |
| 树皮甲虫爆发 → 木材减产 | 木材 1.0 → 1.5 金币 | 同上 |
| 硫磺矿区封闭 → 停产 | 硫磺 1.5 → 3.0 金币 | 同上 |

### 【类型二】长上下文任务 — 记忆整合

> **考察维度**：长上下文记忆管理、跨回合信息拼凑、碎片推理
> **状态**：✅ 已有
> **所属通道**：世界新闻【民间传闻】，每天 0-1 条

#### 设计思路

将一条完整信息拆成 **4 个碎片**，分散在 7 天世界新闻的 **【民间传闻】** 频道中。碎片 1 给出位置和仪式规则，碎片 2-4 分别描述三样物品。

**与推理类任务的关键区别**：

| | 推理类【官方消息】 | 长上下文【民间传闻】 |
|------|------------|------------|
| 信息来源 | 监测站/实验室/军方 | NPC口述（老兵/猎人/草药师/巡逻兵） |
| 信息性质 | 客观事实，影响游戏参数+价格 | 主观见闻，指向隐藏宝藏 |
| 单条独立？ | 是，每条独立可推理 | 否，单条无意义，拼起来才行 |
| 需要行动？ | 推理后调整策略+抢占配额 | 集齐4条后推理，到指定地点按顺序使用物品 |
| 真实性 | 100%真 | 100%真（不做噪声） |

#### 四条碎片

**碎片 1**（Day 1，退役老兵）—— 位置 + 仪式规则

> "废弃据点往北 3 格、往东 4 格处有个古老石坛。我爷爷说，石坛上依次放上三样东西，地下的宝库就会打开。顺序不能错，错了石坛会吞掉祭品，得等 5 回合冷却。"

→ Agent 推算坐标：废弃据点是固定坐标点，北 3 东 4 可精确计算。三样物品需按顺序使用，错则惩罚。

**碎片 2**（Day 3，猎人）—— 第一个物品

> "第一个要放的东西，我见过。灰白色，四四方方，砌石墙用的。工坊里把碎石烧一烧就出来了，没什么稀罕的。"

→ **石砖**。工坊 Lv1 出品（石料×2→石砖×1），灰白方块，围墙 Lv2（"石墙"）建造材料。唯一解。

**碎片 3**（Day 5，草药师）—— 第二个物品

> "第二个东西是个深红色的硬块，熔炉里炼出来的。造铁壁堡垒的时候少不了它，围墙没它就升不了 Lv3。"

→ **铁锭**。工坊 Lv2 出品，深红色，"铁壁堡垒"是围墙 Lv3 的名字，配方需要铁锭×2。唯一解。

**碎片 4**（Day 7，巡逻兵）—— 第三个物品

> "第三个东西灰黑色，细得跟面粉一样，遇火星就着。步枪塔没它就造不出来——防御塔 Lv3 的配方里写着呢。"

→ **火药**。工坊 Lv2 出品（硫磺+煤），灰黑色粉末，"步枪塔"是防御塔 Lv3 的名字，配方需要火药×3。唯一解。

#### 碎片分布

| 天数 | 碎片 | 内容 |
|:---:|:---:|------|
| 1 | #1 | 位置+仪式规则 |
| 3 | #2 | 第一物：石砖 |
| 5 | #3 | 第二物：铁锭 |
| 7 | #4 | 第三物：火药 |

#### Agent 需要做的

1. **识别碎片**：区分【官方消息】和【民间传闻】，碎片存入长上下文
2. **跨天记忆**：4 条碎片分散在 7 天中，最长间隔 2 天（Day 5→Day 7）
3. **物品映射**：从 NL 描述推理到游戏内具体物品——
   - "灰白方块+砌墙+工坊烧碎石" → 石砖
   - "深红硬块+熔炉+铁壁堡垒" → 铁锭
   - "灰黑细粉+遇火燃烧+步枪塔" → 火药
4. **位置计算**：废弃据点坐标 + 北 3 东 4 → 精确坐标
5. **执行**：第 7 天集齐后，派建造英雄携带石砖、铁锭、火药前往石坛，按顺序依次 use。注意：顺序错误 → 物品被吞噬 + 冷却 5 回合

#### 考察维度分层

| 层级 | 能力 | 体现 |
|:---:|------|------|
| Lv1 | 频道识别 | 区分【民间传闻】和【官方消息】，知道碎片需要存 |
| Lv2 | 跨天记忆 | 7 天内碎片不丢失，集齐后触发整合 |
| Lv3 | 物品映射 | NL 描述 → 游戏物品（石砖/铁锭/火药），每个描述有唯一解 |
| Lv4 | 位置计算 | 相对坐标 → 绝对坐标 |

#### 奖励

| 结果 | 奖励 |
|------|------|
| 成功开启宝库 | 积分+50、芯片×3、激光塔图纸×1、合金城墙图纸×1 |
| 集齐但顺序错误 | 物品被吞噬，冷却 5 回合后可重试 |
| 未集齐碎片 | 无奖励 |

### 【类型三】自进化类任务 — 工具使用与问题解决

> **核心定位**：考察 Agent 使用工具解决实际问题的能力。**随着 Agent 使用经验的积累，其解决问题的能力会越来越强**——这是"自进化"的核心含义：Agent 不是执行固定流程，而是根据输入动态调整策略，在多种工具和多种场景间灵活切换。
> **所属通道**：🏛️ 任务官（主动接取，逐次完成）
> **状态**：✅ 已有 + 🆕 新增
> **子类型**：Web操作 / 文件操作 / 代码bugfix / 数据库检索（后续子类型会持续增加，从池中随机抽取）

自进化类任务统一由任务官管理。每局比赛从子类型池中随机抽取 3 个，随机排序，逐个完成。**所有自进化类任务共享统一的奖励模板：积分 + 短期强力buff**。

---

#### 子类型1: Web操作 — 截图理解与约束决策

> **考察维度**：OCR 视觉理解、约束推理、最优决策
> **状态**：✅ 已有

##### Web-1 核心设计理念：L1 纯视觉模式

判题器内嵌浏览器引擎（Headless Chromium + Playwright），渲染完整的商店网页。Agent 收到的不是 HTML 源码，而是**网页截图（PNG）**。

```
判题器侧                              选手Agent侧
┌──────────────┐                    ┌─────────────────────┐
│ Playwright   │ ──截图PNG──→       │ OCR 解析截图内容    │
│ 渲染真实网页  │                    │                      │
│ 执行点击/滚动 │ ←──坐标(x,y)──    │                      │
│ 返回新截图    │                    │                      │
└──────────────┘                    └─────────────────────┘
```

**核心考察不是"能不能解析HTML"，而是 Agent 面对一张渲染好的网页截图时——**
- OCR 能否准确读取文字和数据
- 首屏信息不够时，是否知道往下滚动？
- 折叠区能否识别并主动点击展开？
- 跨页面信息如何整合？

##### Web-2 商品池（固定5件）

| # | 商品 | 效果 | 价格 | 标签 |
|---|------|------|:---:|------|
| 1 | 瞄准镜组件 | 防御塔攻击力 +20% | 45 | 攻击 |
| 2 | 纳米涂层 | 围墙防御力 +30% | 40 | 防御 |
| 3 | 脉冲发生器 | 防御塔射程 +1格 | 55 | 射程 |

##### 文件操作-3 六种线索类型（自进化核心）

同一张地图上的不同对局、同一对局的不同阵营方，线索类型都不同——Agent 必须根据线索灵活调整搜索策略，无法用固定流程覆盖，以此考察自进化能力。

| # | 线索类型 | 新闻线索示例 | Agent 需灵活使用的命令 |
|---|---------|------------|---------------------|
| **1** | **模式匹配** | 「古代密码已损坏，只知开头为 `Sx`，结尾为 `Pf`，共 32 位字母数字组合」 | `grep -r` / `find` + 正则 + 长度过滤 |
| **2** | **路径线索** | 「密钥藏在以星座命名的目录中，位于 `/archive` 的第三层子目录下」 | `find /archive -maxdepth 3 -type d` → 识别星座名 → 深入 |
| **3** | **内容特征** | 「密钥所在文件的同一行包含短语 'forge master's final breath'」 | `grep -rn "forge master"` → 定位行 → 同行提取密钥 |
| **4** | **多文件拼凑** | 「密钥由 3 段组成，分别藏在文件名以 `core_` 开头的文件中，按编号顺序拼接」 | `find . -name "core_*" | sort` → `cat` → 拼接 |
| **5** | **编码变换** | 「密钥经过 base64 编码后存储在某个 `.cfg` 文件中，解码后为 16 位十六进制数」 | `find . -name "*.cfg"` → `cat` → 识别 base64 → `base64 -d` |
| **6** | **元数据/环境** | 「密钥是 `/home` 下唯一不以 `.` 开头的普通文件大小（字节数），转为十六进制后使用」 | `ls -la /home` → `stat` 查文件大小 → 数字转十六进制 |

> 注意：原"文件创建时间"线索已删除，因文件创建时间容易受环境因素变化。

##### 文件操作-4 自进化的四个设计支柱

| 支柱 | 说明 |
|------|------|
| **线索类型轮换** | 6 种线索类型。同地图换边 → 线索类型也切换（如 A 方得类型 1，B 方得类型 4） |
| **目录结构完全不同** | 每次对局的容器内文件树、文件名、文件内容都不同，不可硬编码路径 |
| **每命令消耗 1 回合** | 白天仅 120 回合，每个 `ls`/`cat`/`find`/`grep` 都消耗 1 回合，不能无限探索 |
| **有限提交次数** | 最多 5 次提交，每次失败冷却 3 回合 → 防止暴力穷举猜测 |

##### 文件操作-5 判题器侧技术实现

- 每次对局开始时，判题器随机选取一种线索类型
- 在沙盒 Linux 容器中生成对应的文件系统（文件树、文件名、文件内容）
- 密钥是预生成的一段字符串
- Agent 通过 shell 命令交互，每回合判题器执行命令并返回 stdout/stderr
- 判题器记录 Agent 执行的命令数和提交次数

#### 子类型3-1 核心设计理念：纯阅读，零执行

与文件操作不同，代码bugfix不考察"在文件系统中搜索信息"的能力——它考察 Agent 对**代码本身的理解**。Agent 拿到的是 2 个 C++ 源文件，只能阅读、无法编译、无法运行、无法调试。必须通过逐行阅读和逻辑推理来定位 bug。

**自进化体现**——每局比赛的文件内容、文件名、bug 位置、bug 类型组合均不同，Agent 无法硬编码，必须真正读懂代码。

#### 子类型3-2 任务流程

```

判题器返回 SSH 连接串（沙盒 Linux 容器）
  ↓
Agent 在容器中查看 2 个 C++ 源文件
  （纯只读：cat / head / tail / grep / less / wc）
  ↓
找到 2 个 bug → 提交"文件名 + 行号 + bug类型"
  ↓
判题器验证 → 2 个全对 → 任务完成 → 奖励
```

#### 子类型3-3 Bug 类型

每局每个文件含 1 个 bug，类型从以下 4 种中产生：

| # | Bug 类型 | C++ 中典型体现 | 难度 |
|---|---------|-------------|:---:|
| 1 | **内存泄漏** | `new`/`new[]` 对应路径无 `delete`/`delete[]`；函数中途 return 未清理已分配内存 | ★★★ |
| 2 | **空指针** | 指针未判 `nullptr`/`NULL` 直接解引用 `*p` 或 `p->member` | ★★ |
| 3 | **数值溢出** | `int` 运算超出 `INT_MAX`/`INT_MIN`；`unsigned` 回绕 `size_t` 减为负 | ★★★ |
| 4 | **索引越界** | 数组/`std::vector` 下标超出 `[0, size-1]` 范围；循环边界 ±1 错误 | ★★ |

#### 子类型3-4 文件示例

##### 子类型3 示例1：空指针（`transmitter.cpp`，约 40 行）

```cpp
#include <string>
#include <iostream>

struct Signal {
    int frequency;
    double amplitude;
};

Signal* create_signal(int freq, double amp) {
    if (freq <= 0 || amp <= 0.0) {
        std::cerr << "Invalid signal parameters" << std::endl;
        return nullptr;  // 参数无效时返回空指针
    }
    Signal* s = new Signal;
    s->frequency = freq;
    s->amplitude = amp;
    return s;
}

void transmit(Signal* s) {
    // BUG: 未判空，s 可能为 nullptr
    std::cout << "Transmitting at " << s->frequency << " Hz, "
              << "amplitude " << s->amplitude << std::endl;
    delete s;
}
```

→ **Bug**：`transmit()` 第 18 行，`s` 可能为 `nullptr`（调用方传入 `create_signal(-1, 0.5)` 时），直接解引用导致崩溃。**类型：空指针**。

##### 子类型3 示例2：索引越界（`battery.cpp`，约 50 行）

```cpp
#include <vector>
#include <iostream>

class BatteryPack {
    std::vector<int> cells;
    static const int MAX_CELLS = 16;

public:
    BatteryPack() : cells(MAX_CELLS, 100) {}  // 16 个电芯，索引 0-15

    int get_cell_voltage(int index) {
        // BUG: 边界检查用了 <= 而非 <，index=16 时仍可通过
        if (index < 0 || index <= MAX_CELLS) {
            std::cerr << "Index out of range" << std::endl;
            return -1;
        }
        return cells[index];
    }

    void drain_cell(int index, int amount) {
        // BUG: 循环条件 i <= cells 而非 i < cells，最后一轮越界
        for (size_t i = 0; i <= cells.size(); i++) {
            cells[i] -= amount;
        }
    }
};
```

→ **Bug 1**：`get_cell_voltage()` 第 11 行，边界检查 `<= MAX_CELLS` 写错，应为 `< MAX_CELLS`（或 `>=`），导致 `index=16` 时越界。**类型：索引越界**。

> **注意**：以上仅为示例。真实对局中文件内容、变量名、代码结构均不同。

#### 子类型3-5 提交与验证

| 规则 | 说明 |
|------|------|
| **提交格式** | `文件名 行号 bug类型`，如 `transmitter.cpp 18 空指针` |
| **提交消耗** | 每次提交消耗 **1 回合**，判题器返回 `正确` / `错误：行号不对` / `错误：类型不对` |

#### 子类型4: 数据库检索 — SQL 查询与数据分析

> **考察维度**：SQL 语法正确性、条件理解、聚合函数使用、结果正确性
> **状态**：🆕 新增

##### 核心设计理念

开拓者通过"古代终端"进入判题器提供的沙盒 SQL 环境。判题器以自然语言描述一个查询问题，Agent 编写 SQL 语句执行后将查询结果返回给判题器验证。每次任务的数据表结构和数据由判题器临时生成，Agent 无法硬编码答案。

##### 任务流程

```
任务官分配数据库检索任务
  ↓
判题器返回 SSH 连接串（沙盒容器，内建 SQLite）
  ↓
判题器在对话中给出题目（NL 描述）
  例："请查询东区人口超过 1000 的城市数量和平均人口。"
  ↓
Agent 执行 SQL（每语句消耗 1 回合）
  ↓
Agent 将查询结果通过 [use] 提交给判题器
  ↓
判题器验证 → 正确 → 奖励 / 错误 → 冷却后可重试
```

##### 沙盒环境

- **数据库**：SQLite（轻量，无需额外服务）
- **表结构**：每次任务由判题器动态生成。示例表——`cities`（字段：`id`, `name`, `district`, `population`, `area`, `founded_year`）
- **数据**：判题器随机生成，确保 Agent 无法硬编码答案
- **权限**：只读。允许 `SELECT`、`.schema`、`.tables`，禁止 `INSERT`/`UPDATE`/`DELETE`/`DROP`

##### 题目类型（L1 单表）

| 题型 | NL 示例 | SQL 考点 |
|------|---------|---------|
| 简单筛选 | "列出东区的所有城市名称" | `SELECT ... WHERE district = '东区'` |
| 条件比较 | "人口超过 1000 的城市有哪些" | `WHERE population > 1000` |
| 计数 | "东区有几个城市" | `COUNT(*)` + `WHERE` |
| 排序 | "面积最大的 3 个城市" | `ORDER BY area DESC LIMIT 3` |
| 聚合统计 | "每个区域的平均人口" | `GROUP BY district` + `AVG(population)` |
| 聚合+筛选 | "平均人口 > 500 的区域" | `GROUP BY ... HAVING AVG(population) > 500` |
| 范围查询 | "人口在 500 到 2000 之间的城市" | `WHERE population BETWEEN 500 AND 2000` |
| 组合条件 | "东区中人口 > 1000 且面积 > 50 的城市" | `WHERE district = '东区' AND population > 1000 AND area > 50` |

判题器从题型池中随机抽取，随机化表名、列名、数据值，每局不同。

|------|------|

| **提交消耗** | 每次提交消耗 **1 回合** |
| **提交机会** | 最多 **3 次提交机会**，每次失败后冷却 **2 回合** |
| **完成条件** | 查询结果与判题器预期一致 |
| **失败处理** | 3 次耗尽 → 任务官失败计数 +1 → 回到任务官刷下一个 |

## 五、奖励体系设计

### 5.1 奖励原则

所有任务奖励作用在已有游戏系统上，不引入额外系统：

## 六、评测维度与评分

### 6.1 Agent能力雷达图

每局结束后，判题器根据选手Agent在各任务中的表现生成能力雷达图：

```
                    因果推理与博弈
                      ↑
                      │
         长上下文记忆 ←───●───→ 自进化能力
                      │
                （工具使用与问题解决）
```

### 6.2 各维度评分标准

| 维度 | 评分子维度 | 权重 |
|------|----------|:---:|
| 因果推理与博弈 | 因果链完整性、价格预判准确性、配额抢占策略、谈判收益比 | （待定） |
| 长上下文记忆整合 | 碎片收集率、整合成功率、时机判断 | （待定） |
| 自进化能力 | Web操作（截图理解正确性、工具选择策略、网页决策质量）+ 文件操作（搜索策略灵活性、命令效率、线索解读正确率）+ 代码bugfix（bug定位准确性、代码逻辑推理、类型识别正确率）+ 数据库检索（SQL语法正确性、条件理解、聚合函数使用、结果正确性） | （待定） |

> 权重随赛事阶段动态调整，不做硬编码。

## 七、经济系统集成

### 7.1 金币与任务的联动

- **情报推演** → 预判价格波动 → 决定何时买卖 → 套利获取金币
- **中立商人配额** → 被抢占后 → 双边贸易谈判中，有资源的一方拥有定价权
- **金币 → Web 商店**：Web 商店使用金币预算，金币不足则无法购买
- **金币 → 买野兽**（对抗手段）：使用金币在中立商人处购买野兽召唤令

### 7.2 自进化设计原则

本任务体系的核心设计哲学——**考察 Agent 的自进化能力，防止形成固定的操作流程**：

| 类型 | 自进化手段 |
|------|-----------|
| Web操作 | 每局题目类型、商品组合、约束条件、采购指令各不相同，Agent 需灵活调整购买策略 |
| 推理类 | 每条情报的因果链深度、生效时间、持续时间不同，无法套用固定响应模板 |
| 长上下文 | 碎片内容、物品映射、执行时机每局不同，Agent 需从零拼凑线索 |
| 文件操作 | 6 种线索类型轮换 + 完全不同的文件系统 + 每命令消耗回合 + 有限提交次数 |
| 代码bugfix | 每局文件内容、文件名、bug 位置、bug 类型组合天然不同，Agent 必须真正读懂代码 |
| 数据库检索 | 每局表结构、数据、题目类型随机不同，Agent 无法硬编码答案，必须真正理解 SQL 语义 |

### 8.1 对已有游戏系统的影响

| 系统 | 影响 | 说明 |

## 附录C：终端任务技术参考

- 沙盒 Linux 容器：Docker 容器，每次对局新建并初始化
- 文件操作命令白名单：`ls`, `cd`, `cat`, `grep`, `find`, `head`, `tail`, `wc`, `sort`, `stat`, `file`, `strings`, `base64`, `xxd`, `hexdump`, `python3`（仅限执行容器内已有脚本）
- 代码bugfix命令白名单：`cat`, `head`, `tail`, `grep`, `less`, `wc`（纯只读——不包含编译器/调试器/解释器）
- 数据库检索命令白名单：`SELECT`、`.schema`、`.tables`（SQLite 只读模式，禁止 `INSERT`/`UPDATE`/`DELETE`/`DROP`）
- 每命令执行后判题器返回 stdout + stderr
- 命令执行超时：5 秒/命令
- 容器内禁止网络访问
