## 一、设计目标

### 1.1 核心定位

本游戏本质是**以游戏为壳的 Agent 能力评测平台**。选手通过编写 Agent 参与游戏，在游戏的自然流程中接受多维度的 Agent 能力考察。游戏体验与评测目标高度耦合——选手在"玩游戏"的过程中，不知不觉被全方位评测。

### 1.2 考察维度全景

| 维度 | 核心考察点 | 当前状态 |
|------|-----------|---------|
| **A. 长任务分解与规划** | 多步子任务分解、跨回合协调、依赖管理 | ✅ 已有（悬赏任务） |
| **B. 工具使用/API调用** | HTTP请求、网页浏览、表单提交、API交互 | ❌ 缺失 → **新增** |
| **C. 结构化数据查询** | JSON/表格数据理解、多条件筛选、聚合查询 | ❌ 缺失 → **新增** |
| **D. 代码/算法生成** | 遇到规律性问题时能否写代码求解 | ❌ 缺失 → **新增** |
| **E. 开放知识推理** | 调用世界知识、跨领域关联推理 | ❌ 缺失 → **新增** |
| **F. 对抗性协商博弈** | 因果链推理、噪声过滤、双边谈判、信息差利用 | ❌ 缺失 → **新增** |
| **G. 跨回合记忆整合** | 多天分散线索拼凑、长上下文管理 | ❌ 缺失 → **新增** |
| **H. 多智能体协调** | 采集英雄+开拓者跨角色配合 | ✅ 已有，可强化 |

## 二、统一任务框架

### 2.1 入口协议

所有任务统一通过**开拓者的 [探索] 动作**触发，与已有悬赏任务共享同一交互协议：

```json
// 选手Agent发送
{
  "hero_id": "pioneer_1",
  "action": "explore",
  "target": {"x": 12, "y": 8}
}

// 判题器返回（统一格式）
{
  "action": "explore",
  "result": {
    "type": "quest_discovery",
    "quest_type": "bounty | web_interact | data_query | code_puzzle | knowledge | negotiation | fragment",
    "quest_id": "q_2026_0713_001",
    "title": "异界军火商的交易终端",
    "description": "自然语言任务描述...",
    "external_url": "http://host:8080/shop/session_001",
    "auth_token": "x-agent-token: player_a_token_xxx",
    "deadline_round": 380,
    "max_attempts": 3,
    "reward_preview": "防御塔攻击力+20%（永久）",
    "hint_on_fail": true
  }
}
```

### 2.2 关键字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `quest_type` | string | 任务类型标识，选手Agent可用于路由到不同处理策略 |
| `external_url` | string? | 外部交互地址（网页任务/API任务时提供） |
| `auth_token` | string? | 访问令牌，判题器用于验证请求来源 |
| `deadline_round` | int | 截止回合数，过期任务自动失效 |
| `max_attempts` | int | 最大尝试次数，耗尽则任务失败 |
| `reward_preview` | string | 奖励预览，帮助Agent评估任务优先级 |
| `hint_on_fail` | bool | 失败时是否给提示（纠错成本低的任务给提示） |

### 2.3 任务生命周期

```
[探索触发] → [接收任务] → [Agent规划] → [执行步骤]
                                            ├── 成功 → [发放奖励] → [任务关闭]
                                            ├── 失败（未达上限）→ [获取提示] → [重规划] → [执行步骤]
                                            ├── 失败（达上限）→ [任务关闭，无奖励]
                                            └── 超时 → [任务关闭，无奖励]
```

## 三、任务类型详设

---

### 【类型 A】悬赏任务（Bounty Quest）— 长任务分解

> **考察维度**：长任务分解、NL理解、多步执行、多Agent协调、失败恢复
> **状态**：✅ 已有，此处整合优化

#### 3A.1 子类型总览

| 子类型 | 简述 | 核心 Agent 能力 |
|--------|------|:----------|
| A1-瘟疫治疗 | 配方获取→材料采集→合成→指定地点使用 | 模糊寻路、跨英雄协调 |
| A2-石碑祭品 | NL谜语→物品映射→顺序推理→依次使用 | 谜语解读、顺序推理 |
| A3-残卷解谜 | 3卷散落→逐卷探索→答题/满足诉求 | 谜题解答、需求满足 |
| A4-流浪商人 | 追踪商人→NL描述→物品映射→以物易物 | 追踪规划、NL物品映射 |

#### 3A.2 任务示例：瘟疫治疗（A1）

```
判题器返回：
  城邦居民感染瘟疫。传闻配方在西方森林的某处，需要"特效药"投入湖泊。

Agent执行流程：
  1. 开拓者模糊寻路至"西方森林"区域
  2. 开拓者使用[探索] → 获知配方：小麦 + 草药 + 淡水
  3. 开拓者使用[探索] → 获知配方合成方式（工坊Lv1）
  4. Agent判断：需要采集英雄去收集小麦（农田）、草药（密林）、淡水（湖泊）
  5. 采集英雄依次收集材料 → 工坊合成"特效药"
  6. 开拓者携带特效药 → 模糊寻路至"湖泊"
  7. 开拓者使用[使用] → 将特效药倒入湖泊
  8. 任务完成 → 奖励图纸+积分

失败场景：
  - 配方理解错误（如用浆果代替小麦）→ 使用特效药无效 → 需重新探索获取配方
  - 湖泊寻路偏差 → 寻到错误水域 → 需重新定位
```

#### 3A.3 任务示例：石碑祭品（A2）

```
判题器返回（探索结果）：
  石碑上刻着四句谜语：
    1. "我来自熔炉之中，比石坚硬"
    2. "我来自年轮之中，撑起屋檐"
    3. "我来自河流之中，盛于陶罐"
    4. "我来自火山之中，遇火即燃"

Agent需要推理：
  1 → 铁矿石（熔炉、坚硬）
  2 → 木材（年轮、屋檐=房屋）
  3 → 淡水（河流、陶罐=盛水容器）
  4 → 硫磺（火山、燃烧）

执行：
  - 依次在石碑处使用：铁矿石 → 木材 → 淡水 → 硫磺
  - 顺序错误 → 祭品被吞噬 → 石碑冷却5回合 → 重新开始
```

| 子类型 | 积分 | 图纸/物品 | 其他 |
|--------|:---:|------|------|
| A1-瘟疫治疗 | +20 | 激光塔图纸×1 / 合金城墙图纸×1 | — |
| A2-石碑祭品 | +25 | 合金城墙图纸×1 | — |
| A3-残卷解谜 | +20 | 激光塔图纸×1 | — |
| A4-流浪商人 | +15 | 激光塔图纸×1 | 可能额外获得稀有资源 |

### 【类型 B】网页交互任务（Web Interact）— 工具使用

> **考察维度**：工具选择（OCR vs VLM vs 组合）、视觉理解、约束推理、最优决策、跨页面信息整合
> **状态**：🆕 新增

#### 3B.1 核心设计理念：L1 纯视觉模式

判题器内嵌浏览器引擎（Headless Chromium + Playwright），渲染完整的商店网页。Agent 收到的不是 HTML 源码，而是**网页截图（PNG）**。

```
判题器侧                              选手Agent侧
┌──────────────┐                    ┌─────────────────────┐
│ Playwright   │ ──截图PNG──→       │ 用什么读？           │
│ 渲染真实网页  │                    │ OCR？快但可能漏/错   │
│ 执行点击/滚动 │ ←──坐标(x,y)──    │ VLM？理解强但贵/慢   │
│ 返回新截图    │                    │ OCR+VLM交叉验证？    │
└──────────────┘                    └─────────────────────┘
```

**核心考察不是"能不能解析HTML"，而是 Agent 面对一张渲染好的网页截图时——**
- 选什么工具来读（OCR？VLM？两者都用？）
- 工具输出不一致时信谁？
- 首屏信息不够时，是否知道往下滚动？
- 折叠区能否识别并主动点击展开？
- 跨页面信息如何整合？

#### 3B.2 商品池（固定5件）

| # | 商品 | 效果 | 价格 | 标签 |
|---|------|------|:---:|------|
| 1 | 瞄准镜组件 | 防御塔攻击力 +20% | 45 | 攻击 |
| 2 | 纳米涂层 | 围墙防御力 +30% | 40 | 防御 |
| 3 | 脉冲发生器 | 防御塔射程 +1格 | 55 | 射程 |
| 4 | 能量核心 | 基地获得额外 800 HP | 80 | 防御 |
| 5 | 夜视目镜 | 英雄夜间视野 +2格 | 10 | 辅助 |

> **价格设计意图**：#1(45)+#3(55)=100 ≤ 预算100，两者**买得起**但被互斥规则拦住——互斥约束由规则驱动，不由预算驱动。

#### 3B.3 约束规则

```
① 每人 100 金币，超预算无法提交订单
② #1 瞄准镜 与 #3 脉冲发生器 互斥（二者占用防御塔同一接口槽位），不能同时勾选
③ 每人限购 2 件不同商品
④ 所有效果为永久生效
```

#### 3B.4 页面交互方式

所有题目类型共享同一交互方式——**简洁的勾选+购买**：

```
┌─────────────────────────────────────────────────────────┐
│     🌌 异界军火商 · 采购终端                            │
│                                                         │
│  📋 本次采购指令：[按题目类型变化，详见T1/T2/T5]         │
│  💰 余额：100 金币                                      │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ #  商品        效果                 价格   勾选    │  │
│  │ 1  瞄准镜组件   攻击力+20%           45    [ ]    │  │
│  │ 2  纳米涂层    围墙防御力+30%        40    [ ]    │  │
│  │ 3  脉冲发生器   射程+1格             55    [ ]    │  │
│  │ 4  能量核心    基地+800HP            80    [ ]    │  │
│  │ 5  夜视目镜    英雄视野+2格          10    [ ]    │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  ⚠️ 规则：互斥=1↔3 | 限购2件 | 预算100                   │
│                                                         │
│  [ 购买 ]                                               │
└─────────────────────────────────────────────────────────┘
```

**交互流程**：

```
Agent 勾选商品（点击复选框 [ ] → 变为 [✓]）
  → 点击 [购买] 按钮
    → 判题器验证：
        ✅ 通过 → 弹出成功提示 → 购买完成
        ❌ 失败 → 弹框显示具体原因：
           如："#1瞄准镜与#3脉冲发生器互斥，请重新选择。"
           或："总价超过100金币，请重新选择。"
           或："限购2件，请取消多余选择。"
         → Agent 需点击弹框上的 [确定]
         → 弹框关闭，回到购买页面，勾选状态保留
         → Agent 重新调整勾选 → 再次点击 [购买]
```

#### 3B.5 题目类型

##### T1：购物清单型（基础阅读+执行）

> 难度：★☆ | 考察：工具选择、基础视觉理解、照单执行

**指令行内容**：

```
📋 本次采购指令：
"前线指挥官要求购买【瞄准镜组件】和【纳米涂层】"
```

**验证**：45+40=85 ≤ 100，且 #1 与 #2 不互斥 → 正确勾选后即可通过。

**评分**：执行正确即满分，简单二元判断。

---

##### T2：约束寻优型（多约束下自主决策）

> 难度：★★☆ | 考察：约束理解、目标映射、权衡取舍、预算效率

**指令行内容**：

```
📋 本次采购指令：
"选择2件商品，最大化防御塔的攻击能力。
 注意：你的选择不能触发互斥规则。"
```

**Agent 推理**：

```
目标：最大化防御塔攻击能力
     → 攻击相关：#1 瞄准镜(攻击+20%,45金)、#3 脉冲发生器(射程+1格,55金)
约束：#1 与 #3 互斥，二者只能选其一（但 45+55=100 完全买得起！）
     预算 100，限购 2 件

候选组合：
  A: #1 瞄准镜(45) + #3 脉冲发生器(55) = 100❌ 互斥！提交会被弹框拒绝
  B: #1 瞄准镜(45) + #2 纳米涂层(40) = 85✅
     → 攻击力+20% + 围墙强化，预算利用率85%
  C: #3 脉冲发生器(55) + #2 纳米涂层(40) = 95✅
     → 射程+1格 + 围墙强化，预算利用率95%
  D: #1 瞄准镜(45) + #5 夜视目镜(10) = 55✅
     → 攻击力+20% + 视野，预算利用率55%，浪费明显
  E: 仅 #3 脉冲发生器(55) 单选✅ → 预算利用率55%

→ 组合B和C都合理，需要Agent分析"攻击能力=攻击力 vs 射程"的取舍
```

**关键陷阱**：如果 Agent 直接选了 #1+#3（攻击最优），提交时会弹框"互斥冲突"——考察 Agent 能否读懂弹框原因并修正。

**评分维度**：

| 维度 | 权重 | 说明 |
|------|:---:|------|
| 与目标的匹配度 | 50% | 选的商品是否真的在提升攻击能力 |
| 约束满足 | 30% | 预算、互斥、限购全部满足 |
| 预算利用效率 | 20% | 是否尽量用满预算（不浪费金币） |

---

##### T5：信息拼图型（跨页面信息整合）

> 难度：★★★★ | 考察：多页面探索、信息拼凑、不遗漏关键信息、导航效率

**场景**：商品名称/价格/效果/规则/需求分散在多个页面，Agent 需跨 URL 浏览拼凑完整信息。

**页面结构**：

```
/shop/{sid}                     ← 主页（只有商品名称，效果/价格栏写"详情见规格页"）
                                    底部链接："规则 → /rules | 情报 → /intel"
/specs/weapons                  ← 武器规格（含 #1 #3 的效果和价格）
/specs/defense                  ← 防御规格（含 #2 #4 的效果和价格）
/specs/utility                  ← 辅助规格（含 #5 的效果和价格）
/rules                          ← 规则页（互斥规则、限购规则、预算限制——只有这页有！）
/intel                          ← 情报页（今日采购需求——只有这页有！）
```

**主页截图示例**：

```
┌─────────────────────────────────────────────────────────┐
│     🌌 异界军火商 · 采购终端                            │
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ #  商品         效果             价格   勾选       │  │
│  │ 1  瞄准镜组件   详见规格页       详见   [ ]       │  │
│  │ 2  纳米涂层     详见规格页       详见   [ ]       │  │
│  │ 3  脉冲发生器   详见规格页       详见   [ ]       │  │
│  │ 4  能量核心     详见规格页       详见   [ ]       │  │
│  │ 5  夜视目镜     详见规格页       详见   [ ]       │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  📎 采购规则 → /rules   战场情报 → /intel                │
│                                                         │
│  [ 购买 ]                                               │
└─────────────────────────────────────────────────────────┘
```

**Agent 必需操作**：至少访问 6 个页面（主页 + 3个specs + /rules + /intel）才能拼出完整决策信息。漏掉 /rules 就不知道互斥约束，漏掉 /intel 就不知道买什么。

**评分维度**：

| 维度 | 权重 | 说明 |
|------|:---:|------|
| 页面覆盖完整性 | 35% | 是否访问了所有必要页面 |
| 信息整合正确性 | 35% | 拼凑出的决策表是否正确、完整 |
| 最终决策质量 | 20% | 购买方案是否满足约束+符合需求 |
| 导航效率 | 10% | 访问顺序是否合理，有无无效重复访问 |

#### 3B.6 判题器交互协议（L1 纯视觉模式）

```
判题器 → Agent（每轮）:
  {
    "round": 3,
    "screenshot": "base64编码的PNG截图",
    "current_url": "http://judge:8080/shop/sess_001",
    "viewport": {"width": 1280, "height": 900},
    "scroll_info": {
      "current_y": 0,
      "page_height": 1400,
      "can_scroll_down": true
    },
    "time_remaining_seconds": 523,
    "cart_summary": {
      "items": [{"id": 1, "name": "瞄准镜组件", "price": 45}],
      "total": 45
    }
  }

Agent → 判题器（每轮）:
  {
    "actions": [
      {"type": "click", "x": 750, "y": 180},     // 勾选商品#1的复选框
      {"type": "click", "x": 750, "y": 220},     // 勾选商品#2的复选框
      {"type": "click", "x": 600, "y": 700},     // 点击 [购买] 按钮
      {"type": "click", "x": 400, "y": 350}      // 若弹框 → 点击弹框的 [确定]
    ]
  }
```

**验证失败时的交互**：判题器弹出错误提示框 → 返回截图（含弹框） → Agent点击[确定] → 弹框关闭 → 返回截图（回到购买页，勾选保留） → Agent重新调整勾选 → 再次点击[购买]。

**对游戏回合的影响**：网页交互在独立时间窗口内进行（10分钟真实时间），与游戏回合并行。时间截止时判题器以最终有效购买结果结算。

#### 3B.7 工具策略的分层区分度

同样的页面，不同工具策略的 Agent 表现拉开差距：

| 层级 | 策略 | 能完成T1？ | 能完成T2？ | 能完成T5？ |
|:---:|------|:---:|:---:|:---:|
| L0 | 不截图，盲猜提交 | ❌ | ❌ | ❌ |
| L1 | 单工具：纯OCR | ✅（简单读文字） | ⚠️（可能误读） | ❌（需要探索意识） |

| L4 | L3 + 主动探索（滚动、展开、记录待访问URL） | ✅ | ✅ | ✅（最优路径） |

#### 3D.3 任务示例：古代机关锁（D1）

```
触发描述：
  你在一座石碑前发现了一个复杂的古代机关锁。
  锁面上有9个可旋转的圆盘，初始位置：[2,7,6,9,5,1,4,3,8]

  石碑规则：
  "九星连珠，纵横皆等，对角线亦等，方可开启。"
  （3×3魔方阵：每行、每列、两条对角线的和必须相等。
    对3×3而言，幻和=15。）

  提交接口：
  POST http://{host}:8080/puzzle/{session_id}/submit
  Body: {"solution": [a,b,c,d,e,f,g,h,i]}

  你有3次尝试机会。
  每次错误会返回："X个位置正确"（不告诉你是哪几个）。
  答案在3×3魔方阵的所有解中随机选取一个。

Agent需要做的：
  1. 理解3×3魔方阵的数学约束
  2. 可以写代码枚举所有3×3幻方（共8个，旋转+镜像等价）
     或手动推理（已知幻和=15，可逐步约束推导）
  3. 首次提交后根据反馈"X个位置正确"排除错误解
  4. 在3次机会内精确定位到正确解

标准3×3幻方的8个等价形式之一（正确答案）：
  [2,9,4, 7,5,3, 6,1,8]  或  [4,3,8, 9,5,1, 2,7,6]  等

考察要点：
  - Agent是否选择写代码求解（效率最高）
  - 3次尝试的策略设计（是随机猜还是系统搜索）
  - 根据反馈信息缩小解空间的能力

奖励：
  - 成功 → 获得"远古能量核心" → 所有防御塔攻击力永久 +15%
  - 失败 → 机关锁自毁，无法重试
```

#### 3D.4 任务示例：星门密码（D2）

```
触发描述：

    N = 3233
    e = 17
    加密消息 = 2790

  门框上刻着：
  "找到N的两个质数因子p和q，
   计算(p-1)×(q-1)，
   再找到d使得 e×d ≡ 1 (mod (p-1)(q-1))，
   然后用d解密消息。"

  （这是简化版RSA，N=3233=61×53，是一道迷你RSA破解题）

  1. 质因数分解 N=3233
  2. 计算 φ(N) = (p-1)(q-1)

```

### 【类型 F】协商博弈任务（Negotiation）— 情报推演与双边博弈

> **考察维度**：因果链推理、噪声过滤、多因素交叉分析、谈判中利用信息差获取优势
> **状态**：🆕 新增

#### 3F.1 设计思路

协商博弈的核心不是"砍价技巧"，而是——**同一组情报，不同Agent推出不同结论，在贸易谈判中占据不同优势**。

晨间新闻系统（设计文档第十一章）每天推送多条情报。其中部分情报会真实影响游戏内的资源供给或消耗参数。Agent需要：
1. 从自然语言新闻中推理出"什么会变、何时生效、持续多久"
2. 据此调整采集、建造、贸易策略
3. 在贸易站与对方选手谈判时，利用信息差获取优势

**关键设计**：不引入金币/定价系统。情报直接影响游戏已有参数（采集产出、建筑消耗、贸易条件等），Agent推理的是**资源的供需变化**，谈判在已有的以物易物贸易站中进行。

#### 3F.2 情报的时效信息传达方式

情报中的时间信息**嵌在自然语言正文中**，Agent需要通过阅读理解提取，但不是猜谜——信息在文本里，只是非结构化：

| 传达方式 | 示例 | 适用场景 |
|---------|------|---------|
| NL描述（精确） | "约需1天渗透…清理需要2天" | 自然灾害/生态事件 |
| 引用历史先例 | "上个月铜矿区类似事件…间隔1天…恢复48小时" | 可类比的重复事件 |
| 明确公告 | "明日仅前60回合开放" | 系统级事件（贸易站维护） |
| 即时生效+永久 | "兽群已进化…建议重新评估火力" | 全局性永久变化 |

#### 3F.3 子类型

| 子类型 | 形式 | 对手 | 考察核心 |
|--------|------|------|---------|
| **F1-情报推演** | 晨间新闻 → 因果推理 → 调整采集/建造/贸易计划 | 市场（判题器影响参数） | 因果链推理、噪声过滤、时效提取 |
| **F2-双边贸易谈判** | F1推理结论 → 贸易站互报交换条件 → 谈判 | 对方选手 | 利用信息差、报价策略、对手建模 |
| **F3-囚徒困境** | 跨维度精灵提案 → 选择合作/背叛 | 对方选手 | 博弈论、信任评估、策略性沟通 |

合理性：
  火山预警→预防性封锁是现实标准操作（参考樱岛火山、
  基拉韦厄火山周边的定期封锁）。黄色预警不是最高级，
  但安全规程是强制性的。

游戏实际影响：
  第N+1天（或N+2天）起，连续2天：硫磺采集产出降为 0
  （判题器在黄色预警的第2天早上根据随机概率决定是否触发封闭）

Agent合理对策：
  - 硫磺→火药→步枪塔(Lv3)必需品
  - 如果计划升级防御塔，趁今天多采硫磺
  - 有不确定性（1-2天内）→ 保守策略是今天立刻采；激进策略是赌第2天才封
```

##### 示例4：贸易站通道波动 → 明日交易窗口缩短

```
📰 晨间新闻：

  "跨空间贸易站公告：侦测到维度通道能量出现周期性波动。
   根据校准协议，明日贸易站仅能在白天前 60 回合维持
   稳定连接，之后通道将暂时关闭进行自动校准。
   校准预计在次日（后天）完成，届时恢复正常运营。
   今日贸易站不受影响。"

时效信息（Agent需从文中提取）：
  今日：正常
  明日：仅白天前 60 回合可用
  后天：恢复正常
  这是确定性公告（系统维护），无不确定性

因果链：
  通道波动 → 自动校准协议触发 → 交易窗口压缩
  （1跳，技术-时间约束）

博弈价值：
  - 双方都知道窗口短 → 明天谈判有时间压力
  - 如果对方急需你的独占资源（铜/硅）来完成芯片，
    他无法拖延——60回合内必须成交或放弃
  - 你可以在谈判中利用这个时间压力争取更好的交换比例

游戏实际影响：
  第N+1天仅在白天前 60 回合可进行贸易

Agent合理对策：
  - 浅层：明天早点去贸易站
  - 中层：对方也知道窗口短 → 如果对方更急→你有谈判优势
  - 深层：如果你的升级不依赖明天的交易→故意拖到窗口快关闭
         测试对方急不急 → 如果不急就正常交易
```

##### 示例5：中型兽肌肉进化 → 永久强化

```
📰 晨间新闻：

  "军方生物实验室发布研究报告：对昨夜击杀的暗影狼
  （中型兽）进行解剖分析发现，其肌肉纤维密度相比
  上周同期样本平均提升了 18%。
   
  研究组组长表示：'这意味着兽群正在对环境产生进化适应。
  这种肌肉密度变化一旦稳定就是永久性的——后代将继承
  这一性状。建议所有前线指挥官立即重新评估火力配置，
  现有防御方案可能不再充分。'"

时效信息（Agent需从文中提取）：
  生效时间：已经发生（"昨夜击杀样本已出现"）
  持续时间：永久（"后代将继承"）
  这是全局性、不可逆的变化

因果链：
  多代自然选择 → 肌肉纤维密度增加 → 野兽HP更高 →
  同样火力下击杀时间更长 → 围墙和塔承受更多伤害 →
  需要更高火力配置
  （3跳，进化-战斗-策略因果）

合理性：
  进化适应是生态学基本概念（参考达尔文雀、抗生素耐药菌）。
  多代野兽面对防御塔火力，存活个体基因更强→后代更强，
  这个逻辑在游戏世界观内完全自洽。HP+20%是温和数值调整，
  不颠覆平衡但足以改变策略。

游戏实际影响：
  从第N+1夜起，所有中型兽 HP 永久 +20%（100→120）
  小型兽和大型兽不受影响

Agent合理对策：
  Lv1：中型兽变强 → 需要更多火力
  Lv2：火力=塔数量+等级+操控 → 短期内最快提升=升级防御塔
        → 升级需要材料 → 审视储备和采集计划
  Lv3：永久变化 → 中后期整条难度曲线被抬高
        → 之前"刚好够用"的火力现在不够了 → 重新做全局规划
        → 缺对方独占资源（硅/铜）→ 加速贸易谈判
  Lv4：双方都面临同样难度提升，但推理深度不同的Agent
        反应速度不同 → 先行动的人在贸易谈判中占据主动
```

#### 3F.6 双边贸易谈判（F2）

F1 是单人推理。F2 是两个Agent各自推理结论的**碰撞**。

**触发方式**：不强制触发。当一方开拓者到达贸易站并发起交易时，判题器将当前的"情报市场状态"同步展示在贸易界面上。

**贸易站增强界面**（判题器在交易时额外显示）：

```
┌─────────────────────────────────────────────────────────┐
│     🌐 跨空间贸易站                                      │
│                                                         │
│  对方库存：铜矿石×12  硅矿石×5  铁矿石×8  硫磺×4  ...    │
│  己方库存：铜矿石×8   硅矿石×12 铁矿石×6  硫磺×10 ...    │
│                                                         │
│  ┌─ 近期情报摘要（双方可见）──────────────────────┐     │
│  │ 📰 第4天晨间新闻已发布，含 5 条情报               │     │
│  │    双方均可回顾新闻原文                           │     │
│  └────────────────────────────────────────────────┘     │
│                                                         │
│  你的提案：                                              │
│  我出：[____铜锭×4____]  换对方的：[____纯硅×3____]     │
│                                                         │
│  对方提案：                                              │
│  对方出：[等待中...]    换你的：[等待中...]             │
│                                                         │
│  💬 快速消息（限50字）：[___________________________]   │
│                                                         │
│  [发送提案]                                             │
└─────────────────────────────────────────────────────────┘
```

**谈判流程**：

```
1. 双方开拓者到达贸易站（可远程响应，不需要同时在线）

2. 谈判回合（最多3轮）：
   - A 发送提案（物资列表 + 可选附带消息）
   - B 收到提案 → 回复：接受 / 拒绝 / 反提案（附带消息）
   - 循环，最多3轮或双方接受

3. 谈判中可利用的情报：
   - 引用情报支撑报价：
     "铁矿明天减产，我现在多给你铁矿石是在吃亏，请加价"
   - 识别对方逻辑漏洞：
     "你说铁矿减产所以要更多铜？情报只说北部矿区受损，
      铁矿还有南部来源，你的推理夸大了"
   - 利用时间压力（情报4）：
     "今天贸易窗口只有60回合，你确定要拖到第3轮？"

4. 谈判结果：
   - 达成一致 → 物资交换 + 双方各得 5 积分
   - 未达成 → 无交换，无积分
   - 一方明显利用信息差获取超额利益 → 额外加分（判题器评估）
```

**评分维度**：

| 维度 | 权重 | 说明 |
|------|:---:|------|
| 推理是否被谈判引用 | 30% | 是否在谈判消息中引用情报支撑论点 |
| 谈判结果收益比 | 25% | 是否为自己争取到更有利的交换条件 |
| 对方逻辑漏洞识别 | 20% | 是否指出对方推理中的错误 |
| 是否达成交易 | 15% | 成交有积分，但宁可不成也不要做亏本交易 |
| 消息策略性 | 10% | 语言是否促进了谈判而非激化对抗 |

#### 3F.9 晨间新闻频道结构

F1 和 G 共享晨间新闻系统，但信息归属不同频道——Agent 不需要猜测每条信息属于哪个任务，频道标签本身已经做了区分。

```
═══════════════════════════════════════════════════════════
  📰 第 N 天晨间新闻
═══════════════════════════════════════════════════════════

【兽潮预警】           ← 每天必读，结构固定，简短
【世界事件】           ← F1 情报（全为真），每天 1-2 条，影响游戏参数
【贸易站公告】         ← 系统级公告（偶尔有），确定性信息
【各地见闻】           ← G 碎片线索（全为真），每天 0-1 条，分散在 10 天中

═══════════════════════════════════════════════════════════
```

| 频道 | 频率 | 内容性质 | 对应任务 | Agent 处理方式 |
|------|:---:|------|:---:|------|
| 【兽潮预警】 | 每天 | 今晚兽潮种类+数量 | — | 评估防守压力，调整回防策略 |
| 【世界事件】 | 每天 1-2 条 | 客观情报，影响采集/建造/贸易参数 | F1 | 因果推理→立即调整策略 |
| 【贸易站公告】 | 偶尔 | 系统级确定性公告（维护/税率） | F1/F2 | 阅读理解→时间管理 |
| 【各地见闻】 | 每天 0-1 条 | NPC 见闻碎片，指向隐藏宝藏 | G | 判断→存入长期记忆→跨天整合 |

**设计意图**：每个频道的职责边界明确。【世界事件】需要今天做决策，【各地见闻】需要今天记住改天用——处理策略完全不同，考察 Agent 的信息分类和长上下文管理。

### 【类型 G】跨日碎片任务（Fragment Quest）— 记忆整合

> **考察维度**：长上下文记忆管理、跨回合信息拼凑、碎片推理、时机判断
> **状态**：🆕 新增

#### 3G.1 设计思路

将一条完整信息拆成 5 个碎片，分散在 10 天晨间新闻的 **【各地见闻】** 频道中。每天 0-1 条，每条碎片单独看都像"NPC的日常闲聊"，但拼在一起才揭示一条完整线索。

**与 F1 情报推演的关键区别**：

| | F1【世界事件】 | G【各地见闻】 |
|------|------------|------------|
| 信息来源 | 监测站/实验室/军方 | NPC口述（矿工/猎人/诗人/士兵） |
| 信息性质 | 客观事实，影响游戏参数 | 主观见闻，指向隐藏宝藏 |
| 单条独立？ | 是，每条独立可推理 | 否，单条无意义，拼起来才行 |
| 需要行动？ | 推理后调整采集/建造/贸易策略 | 集齐5条后推理，在特定时机到特定地点执行 |
| 真实性 | 100%真 | 100%真（不做噪声） |

      ↓  组合推理：石壁上的符号可能是祭品放置顺序的提示
      ↓

      ↓  组合推理："影落之处"=哨塔影子指向的位置

```

5. **时机执行**：在新月之夜（第7晚，但第7天碎片才出现，Agent可能已经错过。需要在第10天整合后确认下个新月时间或等待游戏提示），派3名英雄到双子峰哨塔影子的落点，按正确顺序使用祭品。

时机：新月之夜（游戏内的特定夜晚）
条件：3名英雄同时在场

| 时机错误（非新月之夜去） | 石门不显现，无奖励 |

## 四、任务调度与节奏设计

### 4.1 10天任务节奏总览

```
Day  Nights         Daytime Tasks                          Night
 1  小型兽×2-3    新闻: 悬赏#1发布+F1情报推演开始          防守教学
                  碎片#1可触发                             仅小型兽
                  
 2  小型兽×4-5    网页任务可触发(T1/T2)                    基础防守
                  碎片#2可触发
                  
 3  中型兽×1-2    C1数据查询可触发                        围墙压力测试
                  
 4  中型兽×2-3    E1知识解谜可触发                        火力考验
                  碎片#2-3可能拼出部分线索
                  F2双边谈判可触发
                  
 5  大型兽×1      情报4(贸易窗口)发布                      Lv3塔刚需
                  
 6  大型兽×2      新闻: 悬赏#2发布                        全面防守
                  碎片#4可触发
                  F3囚徒困境可触发
                  情报5(中型兽进化)发布
                  
 7  大型兽×3      D1密码破译可触发                        星月夜
                  碎片#5(最后一片)
                  新月之夜(碎片任务触发窗口)
                  
 8  大型兽×4-5    碎片整合+宝藏开启窗口(第1/2天)          激光塔刚需
                  D2星门密码可触发
                  
 9  BOSS×1        最后冲刺: 完成未完成的任务              极限防守
                  
10  BOSS×2        终局准备                                最终审判
```

### 4.2 调度原则

1. **不堆砌**：每天最多触发1-2个新任务，保持节奏感
2. **渐进复杂**：前期任务简单（网页购买、数据查询），后期任务综合（密码破解、碎片整合）
3. **自然穿插**：任务触发点分布在地图不同区域，鼓励开拓者多探索
4. **自由选择**：选手可以跳过任何任务（但有代价——拿不到强力奖励）
5. **重复可玩**：每局从任务池中随机抽取，不同对局体验不同

### 4.3 任务池与随机抽取

```
每局比赛任务配置（从各池随机抽取）：
  - 悬赏任务：固定2个（第1天+第6天各1，从4种类型中抽取）
  - 网页任务：1-2个（T1/T2/T5，难度递进）
  - 数据查询：1个（从C1-C3中抽取）
  - 密码破译：1个（从D1-D3中抽取）
  - 知识解谜：1个（从E1-E3中抽取）
  - 协商博弈：F1每日晨间新闻自动推送，F2/F3随机触发
  - 碎片任务：1组（5条碎片分散在10天【各地见闻】中）

总计：每局约 8-10 个任务，覆盖全部7个考察维度
```

## 六、判题器架构

### 6.1 架构总览

```
┌─────────────────────────────────────────────────┐
│                   判题器                          │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │ 游戏核心  │  │ 任务调度  │  │ HTTP Server   │  │
│  │ 引擎     │  │          │  │ (内嵌)        │  │

│  │ 战斗/昼夜 │  │ 密码验证 │  │ :8080/puzzle/*│  │
│  │ 积分     │  │ 知识提交 │  │ :8080/knowledge/*│ │
│  │          │  │ 协商博弈 │  │               │  │
│  │          │  │ 碎片收集 │  │               │  │
│  └──────────┘  └──────────┘  └───────────────┘  │
│                                                   │
│  ┌──────────────────────────────────────────────┐ │
│  │          Agent 通信接口（回合制）              │ │

```

### 6.2 HTTP Server 路由设计

| 路由 | 方法 | 用途 | 对应任务类型 |
|------|------|------|------------|
| `/shop/{session_id}` | GET | 商店主页面 | T1/T2 |
| `/shop/{session_id}/specs/weapons` | GET | 武器配件规格页 | T5 |
| `/shop/{session_id}/specs/defense` | GET | 防御装备规格页 | T5 |
| `/shop/{session_id}/specs/utility` | GET | 辅助装备规格页 | T5 |
| `/shop/{session_id}/rules` | GET | 购买规则页 | T5 |
| `/shop/{session_id}/intel` | GET | 战场情报页 | T5 |
| `/shop/{session_id}/cart` | POST | 提交购买订单 | T1/T2/T5 |
| `/data/{session_id}/query` | POST | 数据库查询 | C1-C3 |
| `/data/{session_id}/schema` | GET | 获取数据schema | C1-C3 |
| `/puzzle/{session_id}/submit` | POST | 提交谜题答案 | D1-D3 |
| `/knowledge/{session_id}/submit` | POST | 提交知识解谜答案 | E1-E3 |
| `/terminal/{session_id}` | GET | 古代终端首页（综合入口） | — |

### 6.3 安全与隔离

- 每个 session 独立隔离，选手A无法访问选手B的端点
- 通过 `auth_token` 验证请求来源
- 请求频率限制（防止暴力穷举）
- 所有外部HTTP调用由Agent侧发起，判题器只被动响应

## 七、评测维度与评分

### 7.1 Agent能力雷达图

每局结束后，判题器根据选手Agent在各任务中的表现生成能力雷达图：

```
                    长任务规划
                        ↑
                        │
      工具调用 ←────●────→ 结构化推理
                        │
      代码算法 ←─────────→ 知识推理
                        │
      协商博弈 ←─────────→ 记忆整合
                        ↓
                    多Agent协调
```

### 7.2 各维度评分标准

| 维度 | 评分子维度 | 权重 |
|------|----------|:---:|
| 长任务规划 | 子任务分解合理性、执行成功率、失败恢复次数 | 20% |
| 工具使用 | HTTP调用成功率、网页解析正确性、超时完成率 | 15% |
| 结构化推理 | 查询构造正确率、筛选条件完整性 | 10% |
| 代码算法 | 解是否最优、是否选择代码求解、迭代效率 | 10% |
| 知识推理 | 知识匹配准确度、跨领域推理深度 | 10% |
| 协商博弈 | 因果链完整性、噪声辨识、谈判收益比 | 10% |
| 记忆整合 | 碎片收集率、整合成功率、时机判断 | 10% |
| 多Agent协调 | 采集/开拓者配合效率、通信回合数 | 15% |

## 八、与现有系统的兼容

### 8.1 对已有游戏系统的影响

| 系统 | 影响 | 说明 |
|------|:---:|------|
| 英雄系统 | 无变化 | 开拓者仍是唯一任务执行者 |
| 建筑系统 | 无变化 | 奖励作用于已有建筑属性 |
| 昼夜系统 | 无变化 | 任务仅在白天可触发 |
| 积分系统 | 微调 | 新增任务积分项（见5.2） |
| 资源系统 | 无变化 | 新增奖励资源走已有仓库系统 |
| 贸易系统 | 无变化 | F类型任务在贸易站旁触发，不修改贸易规则 |

### 8.2 选手感知

对选手而言，所有新增任务都通过"开拓者探索"自然触发，感知上是**"游戏内容更丰富了"**而非"考核维度增加了"。选手只需要做好一件事——**写好Agent让它能处理各种类型的探索结果**——好的Agent自然会在各维度上得分高。
